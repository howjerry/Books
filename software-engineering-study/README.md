# 實踐軟體工程（第9版）學習筆記

> 寒假作業：追逐新技術固然重要，但在那之前，我想先停下來好好確認——自己是否真的理解工程（Engineering），是否掌握了軟體工程的本質。

---

## 專案目標

這份學習筆記不是要背概念，而是要具備三種能力：

| 能力 | 說明 |
|------|------|
| **選擇能力** | 面對需求、風險、時間、人員、品質，能選對流程、選對設計、選對測試策略，而不是套模板 |
| **表達能力** | 能把需求、架構、設計、品質、風險用共同語言（模型、文件、指標、決策記錄）表達，讓團隊能協作與追溯 |
| **治理能力** | 能建立「可持續交付」的系統：有回饋、有度量、有品質護欄、有變更控制，並能在人與組織的限制下運作 |

## 學習方法

### 每章讀書節奏

#### 1) 章前 5 分鐘：預測
- 這章應該回答什麼問題？
- 我現在的做法是什麼？有哪些痛點？

#### 2) 章中：抓可操作的結論
把每章濃縮成三類輸出：
- **原則（Principle）**：什麼情況下應該這樣做？
- **方法（Practice）**：具體步驟與產物是什麼？
- **護欄（Guardrail）**：什麼跡象代表你做錯了？

#### 3) 章後 15 分鐘：工程化輸出
每章至少留下其中兩個：
- 1 張決策表（何時選 A / 何時選 B）
- 1 份檢查清單（Checklist）
- 1 個反例（Anti-pattern）
- 1 個最小可用模板

## 輸出規格

### 每章固定輸出（必交）

每章 1 頁包含：
1. **本章一句話**：這章最重要的主張是什麼？
2. **3 個關鍵概念**：用自己的話寫
3. **1 個決策點**：什麼情況下要選不同做法？
4. **1 個反模式**：常見錯法與後果
5. **1 個可落地模板 / Checklist**：至少 5 條

### 每部份結束輸出（必交）

- **Part 決策樹（Decision Tree）**
  - 例如：專案類型 → 團隊成熟度 → 風險 → 流程模型選擇 → 品質策略 → 度量方式

### 全書完成輸出（必交）

1. **工程治理地圖（SE OS）**
   - 把流程、需求、設計、測試、風險、度量、配置管理、支援策略串起來
   - 用一頁描述「可持續交付系統」

2. **最小可用文件包**
   - 需求模板
   - 設計模板
   - 測試策略模板
   - 風險清單模板
   - 變更流程模板

## 各部份學習重點

### Part 0：第1章（地基）

**培養能力**：
- 分清楚：軟體本質、工程本質、工藝與科學差異
- 為什麼「軟體危機」至今仍存在？
- 為什麼 SE 是「在限制下管理變更」？

**練習**：
- 寫下近三年最常遇到的 3 個工程問題，對應到：流程、需求、設計、品質、人、管理

---

### Part 1：第2–5章（流程與人）

**培養能力**：
- 流程不是宗教，而是風險管理工具
- 清楚知道什麼情況下用瀑布、增量、螺旋、RUP、Scrum、Kanban、DevOps
- 理解敏捷的本質：縮短回饋迴路、降低交付風險
- 正視：人的行為、溝通、文化會吞掉所有流程設計

**必做輸出**：
- [ ] 流程模型選擇矩陣
- [ ] 團隊行為護欄

---

### Part 2：第6–14章（需求與設計）

**培養能力**：
- 需求不是「寫下來」，而是降低誤解與變更成本
- 模型化是為了：溝通、驗證、分析、設計推導
- 設計不是畫圖，而是：分解、抽象、介面、封裝、權衡
- 架構是「品質屬性」的承諾
- UX 不是美觀，而是任務導向與認知負擔管理

**必做輸出**：
- [ ] 需求三層模板
- [ ] 架構決策記錄 ADR 模板
- [ ] 設計品質檢查清單

---

### Part 3：第15–23章（品質與安全）

**培養能力**：
- 品質不是測試部門的責任，是整個工程系統的設計
- Review 不只是抓 bug，是把知識與標準制度化
- SQA 是「流程 + 護欄 + 度量 + 持續改善」
- Security 不是加密與掃描，而是威脅模型與工程流程內建
- 測試不是寫用例，而是策略

**必做輸出**：
- [ ] 品質策略文件
- [ ] 測試金字塔落地版本
- [ ] 威脅模型最小模板
- [ ] 配置管理流程
- [ ] 核心指標定義

---

### Part 4：第24–27章（專案管理與支援）

**培養能力**：
- PM 的本質：在不確定性下做承諾，並持續調整承諾
- 計畫不是排甘特圖，而是：里程碑、風險、資源、依賴、驗證點
- 風險管理不是填表，而是：提前暴露與降低不確定性
- 支援策略是產品生命週期的一部分

**必做輸出**：
- [ ] 可執行計畫模板
- [ ] 風險登錄表
- [ ] 支援策略文件

---

### Part 5：第28–30章（改善與趨勢與收束）

**培養能力**：
- 流程改善不是「改敏捷儀式」，而是用數據與回饋迴路改系統
- 能辨識「趨勢」與「噪音」
- 收束能力：把 SE 的全貌凝縮成自己的操作系統

**必做輸出**：
- [ ] 改善提案（目標 → 指標 → 介入 → 預期效果 → 風險）
- [ ] 趨勢雷達
