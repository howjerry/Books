## 29.2 AI 與軟體工程

### 29.2.1 AI 輔助開發

**AI Coding Assistants**：

| 工具 | 功能 |
|------|------|
| **GitHub Copilot** | 程式碼補全、生成 |
| **Claude Code** | 程式碼編寫、重構、解釋 |
| **Cursor** | AI-first IDE |
| **Tabnine** | 程式碼補全 |

**AI 如何改變開發**：

```
┌──────────────────────────────────────────────────────────────┐
│                    AI 輔助開發流程                            │
│                                                              │
│  傳統：                                                       │
│  需求 → 設計 → 編碼 → 測試 → 部署                             │
│         ↑      ↑      ↑                                      │
│       人工    人工    人工                                    │
│                                                              │
│  AI 輔助：                                                    │
│  需求 → 設計 → 編碼 → 測試 → 部署                             │
│    ↑     ↑      ↑      ↑      ↑                              │
│   AI    AI     AI     AI     AI                              │
│  協助   協助   生成   生成   自動                              │
└──────────────────────────────────────────────────────────────┘
```

### 29.2.2 AI 對軟體工程師的影響

| 面向 | 變化 |
|------|------|
| **生產力** | 顯著提升（2-10 倍） |
| **技能要求** | 更重視設計、溝通、判斷 |
| **工作內容** | 更多審查、指導、驗證 |
| **學習方式** | AI 作為學習夥伴 |

**新技能需求**：

| 技能 | 說明 |
|------|------|
| **Prompt 工程** | 如何和 AI 溝通 |
| **AI 輸出審查** | 識別 AI 錯誤 |
| **系統設計** | AI 不擅長的領域 |
| **問題定義** | 定義問題比解決重要 |

### 29.2.3 AI 的局限與風險

| 局限 | 說明 |
|------|------|
| **幻覺** | 可能生成錯誤程式碼 |
| **上下文限制** | 難以理解大型系統 |
| **原創性** | 可能抄襲訓練資料 |
| **安全性** | 可能引入漏洞 |

**最佳實踐**：

| 實踐 | 說明 |
|------|------|
| **審查** | 永遠審查 AI 生成的程式碼 |
| **測試** | 用測試驗證正確性 |
| **理解** | 確保理解生成的程式碼 |
| **安全掃描** | 掃描安全漏洞 |

## 29.3 雲原生

### 29.3.1 什麼是雲原生？

**雲原生（Cloud Native）**：充分利用雲端優勢的應用設計和部署方式。

```
┌──────────────────────────────────────────────────────────────┐
│                    雲原生四大支柱                             │
│                                                              │
│  ┌───────────────┐  ┌───────────────┐                       │
│  │   容器化       │  │   微服務      │                       │
│  │  (Containers) │  │ (Microservices│                       │
│  │               │  │              )│                       │
│  └───────────────┘  └───────────────┘                       │
│                                                              │
│  ┌───────────────┐  ┌───────────────┐                       │
│  │   動態編排     │  │   DevOps     │                       │
│  │ (Orchestration│  │              │                       │
│  │              )│  │              │                       │
│  └───────────────┘  └───────────────┘                       │
└──────────────────────────────────────────────────────────────┘
```

### 29.3.2 容器化

| 概念 | 說明 |
|------|------|
| **Docker** | 容器技術標準 |
| **Image** | 容器映像，打包應用 |
| **Container** | 運行中的實例 |
| **Registry** | 映像倉庫 |

**容器的優勢**：

| 優勢 | 說明 |
|------|------|
| **可攜性** | 任何環境都能運行 |
| **隔離性** | 應用間互不干擾 |
| **效率** | 比虛擬機輕量 |
| **一致性** | 開發、測試、生產一致 |

|----------|------|

| **Deployment** | 應用部署管理 |
| **ConfigMap/Secret** | 配置和機密管理 |

### 29.3.4 Serverless

**Serverless**：無伺服器運算，只關注程式碼。

| 特點 | 說明 |
|------|------|
| **事件驅動** | 有請求才執行 |
| **按需計費** | 只付執行時間 |
| **自動擴展** | 無需管理伺服器 |
| **快速部署** | 專注業務邏輯 |

**Serverless 平台**：

| 平台 | 提供商 |
|------|--------|
| AWS Lambda | Amazon |
| Azure Functions | Microsoft |
| Google Cloud Functions | Google |
| Cloudflare Workers | Cloudflare |

## 29.4 平台工程

### 29.4.1 什麼是平台工程？

**平台工程（Platform Engineering）**：建立自助式開發者平台，讓開發者自主完成開發、測試、部署。

```
┌──────────────────────────────────────────────────────────────┐
│                    平台工程的定位                             │
│                                                              │
│                  ┌─────────────────┐                        │
│                  │   產品團隊      │                        │
│                  │  開發者、SRE    │                        │
│                  └────────┬────────┘                        │
│                           │ 使用                             │
│                           ▼                                  │
│                  ┌─────────────────┐                        │
│                  │  內部開發者平台  │                        │
│                  │  (IDP)         │                        │
│                  └────────┬────────┘                        │
│                           │ 整合                             │
│                           ▼                                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │   K8s   │  │   CI/CD │  │  監控   │  │  日誌   │        │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘        │
└──────────────────────────────────────────────────────────────┘
```

### 29.4.2 內部開發者平台（IDP）

| 功能 | 說明 |
|------|------|
| **自助服務** | 開發者自己操作 |
| **黃金路徑** | 標準化的最佳路徑 |
| **抽象複雜度** | 隱藏基礎設施細節 |
| **可觀測性** | 內建監控和日誌 |

**常見平台工具**：

| 工具 | 用途 |
|------|------|
| **Backstage** | 開發者入口 |
| **Crossplane** | 基礎設施管理 |
| **ArgoCD** | GitOps 部署 |
| **Terraform** | IaC 工具 |

### 29.4.3 DevOps vs. 平台工程

| 面向 | DevOps | 平台工程 |
|------|--------|----------|
| **焦點** | 文化和實踐 | 產品和工具 |
| **服務對象** | 運維和開發協作 | 開發者體驗 |
| **交付方式** | 實踐推廣 | 自助平台 |
| **演進** | DevOps 的進化 | 規模化 DevOps |

## 29.5 低程式碼與無程式碼

### 29.5.1 什麼是低程式碼？

**低程式碼（Low-Code）**：使用視覺化介面和最少程式碼開發應用。

| 類型 | 說明 | 範例 |
|------|------|------|
| **No-Code** | 完全不需要寫程式碼 | Bubble, Webflow |
| **Low-Code** | 需要少量程式碼 | OutSystems, Mendix |
| **Pro-Code** | 傳統程式開發 | VS Code |

### 29.5.2 低程式碼的適用場景

| 適合 | 不適合 |
|------|--------|
| 內部工具 | 高複雜度產品 |
| 快速原型 | 高效能需求 |
| 簡單工作流 | 深度客製化 |
| 公民開發者 | 核心系統 |

### 29.5.3 對軟體工程的影響

| 影響 | 說明 |
|------|------|
| **民主化** | 更多人能開發 |
| **效率提升** | 簡單需求快速滿足 |
| **角色變化** | 工程師更專注複雜問題 |
| **治理挑戰** | 影子 IT 風險 |

## 29.6 綠色軟體工程

### 29.6.1 什麼是綠色軟體？

**綠色軟體（Green Software）**：設計和開發對環境影響最小的軟體。

| 原則 | 說明 |
|------|------|
| **碳效率** | 減少碳排放 |
| **能源效率** | 減少能源消耗 |
| **碳感知** | 在低碳時段運行 |
| **硬體效率** | 延長硬體壽命 |

### 29.6.2 為什麼重要？

```
┌──────────────────────────────────────────────────────────────┐
│                    IT 產業的碳足跡                            │
│                                                              │
│   IT 產業佔全球碳排放約 2-4%                                  │
│   相當於航空業的碳排放                                        │
│                                                              │
│   資料中心                      40%                          │
│   █████████████████████████████████████                      │
│                                                              │
│   終端設備                      40%                          │
│   █████████████████████████████████████                      │
│                                                              │
│   網路                          20%                          │
│   ███████████████████                                        │
└──────────────────────────────────────────────────────────────┘
```

### 29.6.3 實踐方法

| 實踐 | 說明 |
|------|------|
| **優化演算法** | 減少計算量 |
| **減少資料傳輸** | 壓縮、快取 |
| **選擇綠色雲端** | 使用再生能源 |
| **延遲非緊急任務** | 在低碳時段運行 |
| **減少雲端資源** | 適當的資源配置 |

## 29.7 安全趨勢

### 29.7.1 Shift Left Security

**左移安全**：將安全融入開發早期。

```
┌──────────────────────────────────────────────────────────────┐
│                    安全左移                                   │
│                                                              │
│  傳統：                                                       │
│  需求 → 設計 → 開發 → 測試 → 部署 → [安全測試]               │
│                                            問題發現太晚       │
│                                                              │
│  左移：                                                       │
│  [安全] → 需求 → 設計 → 開發 → 測試 → 部署                   │
│   ↑威脅建模  ↑安全設計  ↑SAST   ↑DAST                        │
│                                                              │
│   早發現 = 低成本 = 高品質                                    │
└──────────────────────────────────────────────────────────────┘
```

### 29.7.2 DevSecOps

| 實踐 | 說明 |
|------|------|
| **SAST** | 靜態程式碼分析 |
| **DAST** | 動態應用測試 |
| **SCA** | 軟體成分分析 |
| **IaC 掃描** | 基礎設施程式碼掃描 |
| **容器掃描** | 映像安全掃描 |

### 29.7.3 零信任架構

**零信任（Zero Trust）**：永不信任，總是驗證。

| 原則 | 說明 |
|------|------|
| **驗證每個請求** | 不因位置信任 |
| **最小權限** | 只給需要的權限 |
| **假設入侵** | 設計假設已被入侵 |
| **持續驗證** | 不是一次驗證 |
