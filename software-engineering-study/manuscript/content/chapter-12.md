# 第12章　使用者體驗設計

> 「使用者體驗不是讓介面漂亮，而是讓使用者能順利完成任務。最好的介面是使用者不會注意到的介面。」

---

## 12.1 什麼是使用者體驗？

### 12.1.1 UX 的定義

**使用者體驗（User Experience, UX）**：使用者與產品互動時的整體感受，包括有用性、易用性、愉悅感。

UX 包含多個面向：

| 面向 | 說明 | 問題 |
|------|------|------|
| **有用性** | 解決使用者問題 | 這個功能有用嗎？ |
| **易用性** | 容易學習和使用 | 使用者會操作嗎？ |
| **效率** | 快速完成任務 | 需要幾個步驟？ |
| **愉悅感** | 使用時的情緒 | 使用者開心嗎？ |
| **可及性** | 所有人都能使用 | 身障者能用嗎？ |

### 12.1.2 UX vs. UI

| 面向 | UX | UI |
|------|-----|-----|
| 關注點 | 整體體驗 | 視覺呈現 |
| 範圍 | 使用者旅程 | 介面設計 |
| 產出 | 流程、架構 | 畫面、樣式 |
| 目標 | 好用 | 好看 |

**關係**：UI 是 UX 的一部分。好的 UX 需要好的 UI，但好的 UI 不保證好的 UX。

### 12.1.3 為什麼 UX 重要？

| 影響 | 說明 |
|------|------|
| **轉換率** | 好的 UX 提高轉換 |
| **留存率** | 使用者願意再來 |
| **支援成本** | 減少使用者困惑 |
| **開發成本** | 早期發現問題，修改成本低 |
| **品牌形象** | 好體驗 = 好印象 |

## 12.2 UX 設計原則

### 12.2.1 尼爾森的 10 個可用性原則

Jakob Nielsen 提出的經典原則：

| # | 原則 | 說明 |
|---|------|------|
| 1 | **系統狀態可見** | 讓使用者知道發生什麼事 |
| 2 | **符合真實世界** | 使用使用者熟悉的語言 |
| 3 | **使用者控制** | 讓使用者可以取消、返回 |
| 4 | **一致性** | 同樣的東西，同樣的呈現 |
| 5 | **預防錯誤** | 防止錯誤發生比事後修復好 |
| 6 | **辨識優於記憶** | 讓資訊可見，不要讓使用者記 |
| 7 | **彈性與效率** | 支援新手和專家 |
| 8 | **簡約設計** | 只顯示需要的資訊 |
| 9 | **幫助使用者識別錯誤** | 錯誤訊息要清楚、有建議 |
| 10 | **幫助與文件** | 提供需要時可查的幫助 |

### 12.2.2 認知負擔

**認知負擔（Cognitive Load）**：使用者大腦處理資訊的負擔。

**降低認知負擔的方法**：

| 方法 | 說明 | 例子 |
|------|------|------|
| **分塊** | 資訊分成小塊 | 電話號碼分段 |
| **漸進式揭露** | 只顯示當前需要的 | 進階選項隱藏 |
| **預設值** | 提供合理預設 | 表單預填 |
| **一致性** | 減少學習成本 | 按鈕位置固定 |
| **視覺層次** | 突出重要資訊 | 標題大於內文 |

### 12.2.3 Fitts' Law

> 到達目標的時間取決於距離和目標大小。

```
T = a + b × log₂(D/W + 1)

T = 時間
D = 距離
W = 目標寬度
```

**設計啟示**：
- 常用按鈕要大
- 常用按鈕放在容易到達的位置
- 相關按鈕放在一起

### 12.2.4 Hick's Law

> 選項越多，決策時間越長。

```
T = b × log₂(n + 1)

T = 決策時間
n = 選項數量
```

**設計啟示**：
- 減少選項數量
- 將選項分類
- 提供預設選項
- 漸進式揭露

## 12.3 UX 設計流程

### 12.3.1 雙鑽石模型

```
        探索           定義           開發           交付
    ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
    │   發散   │   │   收斂   │   │   發散   │   │   收斂   │
    │         │   │         │   │         │   │         │
   ◄           ►◄           ► ◄           ►◄           ►
    │ 發現問題 │   │ 定義問題 │   │ 發想解法 │   │ 實作解法 │
    │         │   │         │   │         │   │         │
    └─────────┘   └─────────┘   └─────────┘   └─────────┘

      問題空間                      解決方案空間
```

### 12.3.2 使用者研究

| 方法 | 說明 | 適用時機 |
|------|------|----------|
| **訪談** | 深入理解使用者 | 探索階段 |
| **觀察** | 看使用者實際行為 | 探索階段 |
| **問卷** | 收集大量數據 | 驗證假設 |
| **使用性測試** | 測試產品可用性 | 設計後 |
| **A/B 測試** | 比較兩個版本 | 優化階段 |
| **數據分析** | 分析使用行為 | 持續 |

### 12.3.3 Persona（人物誌）

用虛構但基於研究的人物代表目標使用者。

```
┌─────────────────────────────────────────────┐
│                  小明                        │
│           資深軟體工程師，32歲                │
├─────────────────────────────────────────────┤
│ 背景：                                       │
│ - 在科技公司工作 8 年                         │
│ - 每天用 IDE 8+ 小時                         │
│ - 重視效率，討厭重複工作                      │
├─────────────────────────────────────────────┤
│ 目標：                                       │
│ - 快速完成開發任務                            │
│ - 減少上下文切換                              │
│ - 保持程式碼品質                              │
├─────────────────────────────────────────────┤
│ 痛點：                                       │
│ - 工具太多，切換麻煩                          │
│ - 重複的樣板程式碼                            │
│ - 文件過時                                   │
├─────────────────────────────────────────────┤
│ 偏好：                                       │
│ - 鍵盤快捷鍵 > 滑鼠                           │
│ - CLI > GUI                                 │
│ - 簡潔 > 華麗                                │
└─────────────────────────────────────────────┘
```

### 12.3.4 使用者旅程地圖

記錄使用者完成任務的完整過程。

```
階段:    發現    →    評估    →    購買    →    使用    →    推薦
         │           │           │           │           │
行動:  搜尋商品    比較功能    加入購物車   學習使用    分享心得
         │           │           │           │           │
想法:  "有這個嗎?" "哪個好?"   "價格OK"   "怎麼用?"   "很棒!"
         │           │           │           │           │
情緒:    😐          🤔          😊          😕→😊       😄
         │           │           │           │           │
接觸點: 搜尋引擎   產品頁面    結帳頁面    說明文件    社群媒體
         │           │           │           │           │
機會:  SEO 優化   比較表格    簡化結帳    引導教學    分享獎勵
```

## 12.4 介面設計模式

### 12.4.1 導航模式

| 模式 | 適用 | 例子 |
|------|------|------|
| **頂部導航** | 主要功能少（< 7） | 網站主選單 |
| **側邊欄** | 功能多、層級深 | 後台管理 |
| **底部標籤** | 行動裝置、主要功能 | App 主導航 |
| **麵包屑** | 層級導航 | 電商分類 |
| **搜尋** | 內容多 | 文件網站 |

### 12.4.2 表單設計

**原則**：

| 原則 | 說明 |
|------|------|
| **減少欄位** | 只問必要的 |
| **分組** | 相關欄位放一起 |
| **標籤清楚** | 使用者知道填什麼 |
| **即時驗證** | 錯誤立即回饋 |
| **預設值** | 減少輸入 |
| **行動呼籲** | 按鈕文字清楚 |

**錯誤處理**：

```
❌ 不好的錯誤訊息
"錯誤：無效的輸入"

✅ 好的錯誤訊息
"請輸入有效的電子郵件地址（例如：name@example.com）"
```

### 12.4.3 回饋模式

| 類型 | 用途 | 例子 |
|------|------|------|
| **Loading** | 等待時 | 轉圈、骨架屏 |
| **成功** | 操作成功 | 綠色訊息 |
| **錯誤** | 操作失敗 | 紅色訊息 + 解法 |
| **確認** | 危險操作前 | "確定要刪除嗎？" |
| **進度** | 長時間操作 | 進度條 |

### 12.4.4 空狀態設計

當沒有資料時，不要顯示空白：

```
❌ 空白頁面

✅ 空狀態設計
┌─────────────────────────────────────┐
│                                     │
│            📭                       │
│                                     │
│      你還沒有任何訂單                 │
│                                     │
│   開始購物，享受專屬優惠！            │
│                                     │
│       [ 開始購物 ]                   │
│                                     │
└─────────────────────────────────────┘
```

## 12.5 可及性（Accessibility）

### 12.5.1 什麼是可及性？

**可及性（Accessibility, a11y）**：讓所有人都能使用產品，包括身心障礙者。

常見障礙類型：

| 類型 | 說明 | 設計考量 |
|------|------|----------|
| **視覺** | 視障、色盲 | 對比度、螢幕閱讀器 |
| **聽覺** | 聽障 | 字幕、視覺提示 |
| **動作** | 無法使用滑鼠 | 鍵盤支援 |
| **認知** | 閱讀困難 | 簡單語言、清晰結構 |

### 12.5.2 WCAG 指南

Web Content Accessibility Guidelines（WCAG）的四個原則：

| 原則 | 說明 |
|------|------|
| **可感知** | 資訊可以被感知（看、聽） |
| **可操作** | 介面可以被操作（鍵盤、觸控） |
| **可理解** | 內容可以被理解 |
| **穩健** | 相容各種輔助技術 |

### 12.5.3 實用技巧

| 技巧 | 做法 |
|------|------|
| **替代文字** | 圖片加 alt 屬性 |
| **對比度** | 文字對比度 ≥ 4.5:1 |
| **鍵盤支援** | 所有功能可用鍵盤操作 |
| **焦點指示** | 清楚的焦點樣式 |
| **語意標籤** | 使用正確的 HTML 標籤 |
| **標籤關聯** | 表單標籤與輸入框關聯 |

## 12.6 使用性測試

### 12.6.1 什麼是使用性測試？

邀請真實使用者完成任務，觀察他們遇到的問題。

### 12.6.2 測試流程

1. **準備**
   - 定義測試目標
   - 設計測試任務
   - 招募參與者（5-8 人）

2. **執行**
   - 解釋測試目的
   - 請使用者完成任務
   - 觀察並記錄
   - 請使用者思考出聲

3. **分析**
   - 彙整問題
   - 分類嚴重程度
   - 提出改善建議

### 12.6.3 測試任務設計

```
❌ 不好的任務
"測試搜尋功能"

✅ 好的任務
"你想買一雙跑步鞋，請找到一雙價格在 2000 元以下的跑步鞋，並加入購物車。"
```

**任務原則**：
- 具體情境
- 明確目標
- 不暗示操作方式

### 12.6.4 嚴重程度分類

| 等級 | 說明 | 優先級 |
|------|------|--------|
| **致命** | 無法完成任務 | 必須修復 |
| **嚴重** | 造成很大困擾 | 優先修復 |
| **中等** | 影響效率 | 應該修復 |
| **輕微** | 小困擾 | 可以修復 |
| **建議** | 可以更好 | 考慮修復 |

## 12.7 本章總結

### 核心概念

| 概念 | 一句話解釋 |
|------|-----------|
| UX | 使用者與產品互動的整體體驗 |
| 認知負擔 | 使用者大腦處理資訊的負擔 |
| Fitts' Law | 目標大且近，操作快 |
| Hick's Law | 選項少，決策快 |
| Persona | 虛構的目標使用者代表 |
| 使用者旅程 | 使用者完成任務的完整過程 |
| 可及性 | 讓所有人都能使用 |

### UX 設計檢查清單

| 項目 | 問題 |
|------|------|
| 系統狀態 | 使用者知道發生什麼事嗎？ |
| 認知負擔 | 資訊量適當嗎？ |
| 一致性 | 介面一致嗎？ |
| 錯誤處理 | 錯誤訊息清楚嗎？ |
| 可及性 | 鍵盤可操作嗎？對比度夠嗎？ |
| 空狀態 | 沒有資料時怎麼顯示？ |
