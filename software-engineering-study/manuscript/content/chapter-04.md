# 第4章　推薦的流程模型

> 「最好的流程是你願意遵守、能夠改進、並且適合情境的那一個。」

---

## 4.1 沒有銀彈，只有適配

經過前兩章的學習，你應該已經理解：

1. 流程模型有很多種（瀑布、增量、演化、敏捷...）
2. 每種模型都有適用情境
3. 敏捷不是萬能的，瀑布也不是過時的

但你可能還在問：**那我到底該用哪一個？**

這一章的目標是給你一個**決策框架**——不是告訴你「用這個」，而是幫助你問對問題、做出適合你情境的選擇。

---

## 4.2 流程選擇的六個維度

選擇流程時，需要評估以下六個維度：

### 維度一：需求穩定性

**問題**：需求有多穩定？變更頻率如何？

| 程度 | 特徵 | 適合的流程 |
|------|------|-----------|
| 高穩定 | 需求在開始時可完全確定，變更罕見 | 瀑布、V 模型 |
| 中穩定 | 核心需求穩定，細節可能調整 | 增量、RUP |
| 低穩定 | 需求會持續演變，變更是常態 | 敏捷、演化 |

### 維度二：技術風險

**問題**：技術方案有多大的不確定性？

| 程度 | 特徵 | 適合的流程 |
|------|------|-----------|
| 低風險 | 使用成熟技術，團隊經驗豐富 | 任何流程 |
| 中風險 | 部分新技術，需要學習和驗證 | 增量、敏捷 |
| 高風險 | 大量新技術，可行性不確定 | 螺旋、原型 |

### 維度三：回饋可得性

**問題**：多容易獲得真正的使用者回饋？

| 程度 | 特徵 | 適合的流程 |
|------|------|-----------|
| 容易 | 使用者可以頻繁參與、試用 | 敏捷、原型 |
| 中等 | 定期可以獲得回饋，但有延遲 | 增量 |
| 困難 | 只能在最後才能獲得回饋 | 瀑布（但要投資早期驗證） |

### 維度四：合規與文件要求

**問題**：有多嚴格的合規、審計、文件要求？

| 程度 | 特徵 | 適合的流程 |
|------|------|-----------|
| 嚴格 | 需要完整文件、審計軌跡、認證 | 瀑布、V 模型、RUP |
| 中等 | 需要關鍵文件，但有彈性 | 增量、帶文件的敏捷 |
| 寬鬆 | 最小文件，結果導向 | 敏捷、XP |

### 維度五：團隊特性

**問題**：團隊的規模、分佈、成熟度如何？

| 特性 | 適合的流程 |
|------|-----------|
| 小團隊（< 10人）、共處 | 敏捷（Scrum、XP） |
| 中型團隊、部分分散 | 增量、Scrum of Scrums |
| 大型團隊、高度分散 | RUP、SAFe、帶協調的增量 |
| 新手團隊 | 有明確指導的流程 |
| 資深自組織團隊 | 敏捷、Kanban |

### 維度六：專案類型

**問題**：這是什麼類型的專案？

| 類型 | 特徵 | 適合的流程 |
|------|------|-----------|
| 全新產品 | 需求不明、需要探索 | 敏捷、原型、精實 |
| 產品迭代 | 基於現有產品增加功能 | 敏捷、增量 |
| 系統整合 | 整合多個系統、介面明確 | 增量、V 模型 |
| 遺留系統現代化 | 高風險、需要漸進 | 絞殺者模式、增量 |
| 嵌入式/安全關鍵 | 嚴格驗證、難以更新 | V 模型、瀑布 |
| 維運/支援 | 大量小任務、緊急插件 | Kanban |

## 4.3 流程選擇決策樹

基於上述維度，這裡提供一個簡化的決策樹：

```
開始
  │
  ▼
需求穩定嗎？
  │
  ├─ 是 ──► 有嚴格合規要求嗎？
  │            │
  │            ├─ 是 ──► 瀑布 / V 模型
  │            │
  │            └─ 否 ──► 增量模型
  │
  └─ 否 ──► 能獲得頻繁回饋嗎？
               │
               ├─ 是 ──► 技術風險高嗎？
               │            │
               │            ├─ 是 ──► 螺旋 / 原型 + 敏捷
               │            │
               │            └─ 否 ──► 敏捷（Scrum / XP）
               │
               └─ 否 ──► 增量 + 早期原型驗證
```

### 快速參考表

| 情境 | 推薦流程 |
|------|----------|
| 需求穩定 + 嚴格合規 | **瀑布** 或 **V 模型** |
| 需求穩定 + 彈性合規 | **增量模型** |
| 需求不穩定 + 可頻繁回饋 + 低風險 | **Scrum** 或 **XP** |
| 需求不穩定 + 可頻繁回饋 + 高風險 | **螺旋** + **敏捷** |
| 需求不穩定 + 回饋困難 | **增量** + **早期原型** |
| 維運/支援工作 | **Kanban** |
| 大規模敏捷 | **SAFe** 或 **LeSS** |

## 4.4 混合流程：務實的選擇

現實中，很少有專案完美符合單一流程。**混合流程**（Hybrid Process）往往是更務實的選擇。

### 4.4.1 常見的混合模式

**模式一：Water-Scrum-Fall**

```
需求階段（瀑布）
    ↓
開發階段（Scrum）
    ↓
部署階段（瀑布）
```

適用於：
- 需求需要正式簽核才能開始
- 部署需要正式審批流程
- 開發階段可以敏捷

**模式二：敏捷 + 合規文件**

在敏捷流程中增加必要的文件產出：
- Sprint 0：產出架構文件
- 每個 Sprint：更新需求追溯矩陣
- 發布前：產出測試報告

適用於：
- 需要合規文件，但開發可以敏捷
- 醫療、金融等受監管行業

**模式三：原型 + 增量**

```
探索階段（原型）
    ↓
確認需求
    ↓
開發階段（增量）
```

適用於：
- 需求不明確，需要先探索
- 探索後需求會穩定下來

**模式四：Scrumban**

結合 Scrum 的節奏和 Kanban 的流動：
- Sprint 提供計畫和回顧的節奏
- WIP 限制管理流動
- 可以接受緊急插件

適用於：
- 有產品開發也有維運工作的團隊
- 需要節奏，但也需要彈性

### 4.4.2 混合的原則

混合流程不是隨意拼湊。好的混合遵循以下原則：

| 原則 | 說明 |
|------|------|
| **目的清晰** | 每個元素都有明確的目的 |
| **介面明確** | 不同階段/實踐之間的銜接清晰 |
| **一致性** | 核心價值觀一致，不自相矛盾 |
| **可調整** | 根據回饋持續改進混合方式 |

**錯誤的混合**：
- ❌ 形式上敏捷，骨子裡瀑布
- ❌ 兩邊的壞處都有，好處都沒有
- ❌ 為了「看起來現代」而敏捷

**正確的混合**：
- ✅ 根據情境選擇適合的元素
- ✅ 明確知道為什麼這樣混合
- ✅ 持續評估和調整

## 4.5 流程演進路徑

### 4.5.1 從瀑布到敏捷

如果組織目前使用瀑布，想漸進轉向敏捷：

```
階段 1：縮短週期
├── 將大專案拆成多個增量
├── 每個增量 2-3 個月
└── 每個增量都有可交付成果

階段 2：增加回饋
├── 每個增量結束時做 Demo
├── 收集使用者回饋
└── 調整下個增量的計畫

階段 3：引入敏捷實踐
├── 在增量內使用 Sprint
├── 引入 Daily Standup
└── 引入 Retrospective

階段 4：深化敏捷
├── 引入技術實踐（TDD、CI）
├── 自組織團隊
└── 持續交付
```

### 4.5.2 從混亂到有序

如果目前沒有明確流程（「隨便做」）：

```
階段 1：視覺化
├── 建立看板，視覺化工作流程
├── 限制 WIP
└── 建立「完成」的定義

階段 2：建立節奏
├── 每週/雙週固定回顧
├── 每天簡短同步
└── 定期展示成果

階段 3：建立實踐
├── 版本控制
├── 程式碼審查
└── 自動化測試

階段 4：持續改進
├── 度量關鍵指標
├── 根據數據調整
└── 實驗新實踐
```

## 4.6 流程不是終點

### 流程的目的

流程的目的不是遵守流程，而是：
1. **降低風險**：提早發現問題
2. **提高效率**：減少浪費和返工
3. **確保品質**：內建品質，而非事後檢測
4. **促進協作**：讓團隊有共同語言
5. **持續改進**：有基礎可以改進

### 流程成熟度

隨著團隊成熟，對流程的依賴會改變：

| 階段 | 特徵 | 需要的流程 |
|------|------|-----------|
| 初始 | 混亂、英雄主義 | 建立基本流程 |
| 可重複 | 有流程但依賴個人 | 文件化、可傳承 |
| 已定義 | 流程標準化 | 度量、優化 |
| 量化管理 | 用數據管理流程 | 預測、優化 |
| 持續優化 | 持續改進內建在文化中 | 最小化流程 |

**關鍵洞察**：最高成熟度不是「最多流程」，而是「最少但足夠的流程」。

## 4.7 本章總結

### 核心概念

| 概念 | 一句話解釋 |
|------|-----------|
| 六個選擇維度 | 需求穩定性、技術風險、回饋可得性、合規要求、團隊特性、專案類型 |
| 決策樹 | 根據情境特徵，系統性地選擇流程 |
| 混合流程 | 務實地結合多種流程的元素 |
| 流程演進 | 漸進式地改進流程，而非一步到位 |
| 流程目的 | 降低風險、提高效率、確保品質、促進協作、持續改進 |

### 實踐指引

1. **評估情境**：用六個維度評估你的專案
2. **選擇基礎**：用決策樹選擇基礎流程
3. **務實調整**：根據實際限制混合或調整
4. **漸進改進**：不要追求完美，持續小步改進
5. **度量效果**：追蹤關鍵指標，驗證流程效果
