# 第5章　軟體工程的人因面向

> 「軟體是由人寫的，給人用的，在人的組織中開發的。忽略人的因素，再好的流程也會失敗。」

---

## 5.1 為什麼人的因素如此重要？

在前面幾章，我們學習了流程模型、敏捷方法、流程選擇。但有一個殘酷的事實：

**再完美的流程，遇到人的問題，都會失敗。**

### 軟體專案失敗的真正原因

根據多項研究，軟體專案失敗的主要原因並非技術：

| 排名 | 失敗原因 | 類型 |
|------|----------|------|
| 1 | 需求不完整或持續變更 | 溝通 |
| 2 | 缺乏使用者參與 | 溝通 |
| 3 | 資源不足 | 管理 |
| 4 | 不切實際的期望 | 溝通/管理 |
| 5 | 缺乏管理層支持 | 組織 |
| 6 | 需求與規格變更 | 溝通 |
| 7 | 缺乏規劃 | 管理 |
| 8 | 不再需要該系統 | 溝通 |

**觀察**：前 8 個原因中，只有「資源不足」和「缺乏規劃」勉強算管理問題，其餘都是**溝通和人的問題**。

### 技術問題 vs. 人的問題

| 類型 | 特徵 | 解決方式 |
|------|------|----------|
| 技術問題 | 有明確定義、可重複驗證 | 工程方法、工具 |
| 人的問題 | 模糊、情境依賴、涉及情緒 | 溝通、信任、文化 |

技術問題往往可以「解決」，人的問題只能「管理」。

## 5.2 團隊結構與動態

### 5.2.1 團隊規模的影響

**Brooks 法則**：
> 「向一個已經延遲的專案增加人力，只會讓它更延遲。」

原因：溝通開銷隨人數指數增長。

| 團隊大小 | 溝通路徑數 | 協調難度 |
|----------|-----------|----------|
| 3 人 | 3 | 低 |
| 5 人 | 10 | 中 |
| 10 人 | 45 | 高 |
| 20 人 | 190 | 極高 |

**溝通路徑公式**：n(n-1)/2

**實際意義**：
- 小團隊（5-7 人）效率最高
- 大專案應拆成多個小團隊
- 增加人力要考慮溝通成本

### 5.2.2 團隊結構模式

**模式一：民主式（Democratic / Egoless）**

```
    ○ ─── ○
   / \   / \
  ○   ○ ○   ○
```

特點：
- 成員平等參與決策
- 程式碼集體所有
- 適合創新、探索性專案

優點：知識共享、成員成長
缺點：決策慢、可能缺乏方向

**模式二：主程式師式（Chief Programmer）**

```
        ○ 主程式師
       /|\
      / | \
     ○  ○  ○ 支援團隊
```

特點：
- 主程式師負責關鍵決策和核心程式碼
- 支援團隊協助次要工作
- 適合緊急、有明確方向的專案

優點：決策快、一致性高
缺點：依賴個人、知識集中

**模式三：Scrum 團隊**

```
    PO ←→ 團隊 ←→ SM
           ↕
        利害關係人
```

特點：
- 跨功能、自組織
- PO 決定「做什麼」，團隊決定「怎麼做」
- SM 保護團隊、移除障礙

優點：自主性高、適應變化
缺點：需要成熟的團隊

### 5.2.3 團隊發展階段

Tuckman 的團隊發展模型：

| 階段 | 特徵 | 領導者角色 |
|------|------|-----------|
| **形成期**（Forming） | 客氣、觀望、尋找定位 | 指導、建立規則 |
| **風暴期**（Storming） | 衝突、爭論、挑戰權威 | 調解、引導、堅持原則 |
| **規範期**（Norming） | 建立共識、合作開始 | 支持、授權 |
| **績效期**（Performing） | 高效協作、自主運作 | 授權、移除障礙 |
| **解散期**（Adjourning） | 專案結束、成員離開 | 慶祝、知識傳承 |

**關鍵洞察**：
- 風暴期是正常的，不要迴避衝突
- 每次成員變動，團隊可能回到早期階段
- 績效期需要時間建立，不能急

## 5.3 溝通：軟體工程的核心活動

### 5.3.1 溝通的挑戰

軟體開發中的溝通特別困難，因為：

1. **不可見性**：軟體看不見，每個人腦中的「系統」不同
2. **專業術語**：技術人員和業務人員說不同的語言
3. **隱性知識**：很多知識難以言傳
4. **假設差異**：每個人有不同的背景假設

### 5.3.2 溝通模型

```
發送者 → 編碼 → 訊息 → 通道 → 解碼 → 接收者
                  ↑              ↓
                  └─── 回饋 ←────┘
                        ↑
                      噪音
```

溝通失敗可能發生在每個環節：
- **編碼失敗**：說不清楚
- **通道問題**：郵件被忽略、會議時間不對
- **解碼失敗**：理解不同
- **缺乏回饋**：不知道對方是否理解

### 5.3.3 有效溝通的實踐

**實踐一：確認理解**

不要假設對方理解了。用這些方式確認：
- 「讓我確認一下，你的意思是...？」
- 「你能舉個例子嗎？」
- 「如果我理解正確，當 X 發生時，系統應該...」

**實踐二：視覺化**

把抽象的概念變成可見的：
- 畫圖（架構圖、流程圖、草圖）
- 原型（可以點擊的 mockup）
- 例子（具體的 scenario）

**實踐三：選擇適當的通道**

| 溝通類型 | 適合的通道 | 原因 |
|----------|-----------|------|
| 複雜問題討論 | 面對面、視訊 | 高頻寬、即時回饋 |
| 日常同步 | Standup、即時訊息 | 快速、低成本 |
| 需要記錄的決策 | 文件、郵件 | 可追溯 |
| 非緊急通知 | 郵件、公告 | 非同步、不打斷 |

**實踐四：傾聽**

> 「大多數人不是為了理解而聽，而是為了回應而聽。」

主動傾聽的技巧：
- 不打斷
- 提問以確認理解
- 複述對方的觀點
- 注意非語言訊號

## 5.4 衝突管理

### 5.4.1 衝突是正常的

在軟體專案中，衝突無可避免：
- 技術方案的爭論
- 優先級的分歧
- 資源的競爭
- 個性的差異

**關鍵**：衝突本身不是問題，如何處理衝突才是問題。

### 5.4.2 衝突處理風格

Thomas-Kilmann 衝突模型：

```
        高
    競爭 ──────── 合作
關   │           │
注   │   妥協    │
自   │           │
己   │           │
    迴避 ──────── 遷就
        低
        低 ──────── 高
          關注他人
```

| 風格 | 特徵 | 適用情境 |
|------|------|----------|
| **競爭** | 堅持己見 | 緊急情況、重要原則 |
| **合作** | 尋求雙贏 | 重要議題、需要承諾 |
| **妥協** | 各退一步 | 時間有限、勢均力敵 |
| **迴避** | 不處理 | 小事、需要冷靜 |
| **遷就** | 讓步 | 關係重要、自己錯了 |

**最佳實踐**：
- 重要議題用**合作**
- 技術爭論用**合作**或**妥協**
- 不要總是**迴避**，問題會累積
- 不要總是**競爭**，會破壞關係

### 5.4.3 建設性衝突 vs. 破壞性衝突

| 類型 | 特徵 | 結果 |
|------|------|------|
| 建設性 | 聚焦問題、尊重人、尋求解決 | 更好的決策、團隊成長 |
| 破壞性 | 人身攻擊、情緒化、輸贏思維 | 關係破裂、士氣下降 |

**將衝突導向建設性的技巧**：
1. **對事不對人**：「這個方案有風險」而非「你的方案很爛」
2. **探詢意圖**：「幫我理解你為什麼這樣想」
3. **尋找共同點**：「我們都同意 X 很重要，對吧？」
4. **提出選項**：「我們有哪些選擇？」

## 5.5 團隊文化

### 5.5.1 文化是什麼？

> 「文化是『這裡的做事方式』。」

文化包含：
- **價值觀**：什麼是重要的？
- **規範**：什麼行為被接受/不被接受？
- **假設**：不言而喻的共識

### 5.5.2 高效能團隊的文化特徵

Google 的 Project Aristotle 研究發現，高效能團隊有以下特徵：

| 特徵 | 說明 |
|------|------|
| **心理安全** | 可以承認錯誤、提出問題，不怕被嘲笑 |
| **可靠性** | 成員信守承諾、完成工作 |
| **結構與清晰** | 角色、計畫、目標明確 |
| **意義** | 工作對個人有意義 |
| **影響** | 工作對世界有影響 |

**最重要的是心理安全**——如果成員不敢說真話，所有其他都無從談起。

### 5.5.3 建立心理安全

**領導者的行為**：
- 承認自己的錯誤
- 鼓勵提問，不嘲笑「笨問題」
- 感謝提出壞消息的人
- 失敗時聚焦學習，而非責備

**團隊的實踐**：
- Blameless Postmortem：出問題時分析系統，不責備個人
- 鼓勵實驗：允許失敗，從失敗中學習
- 定期回顧：創造安全討論問題的空間

### 5.5.4 警惕有毒文化

有毒文化的警訊：
- 🚩 怪罪文化：出問題就找人負責
- 🚩 英雄文化：靠個人英雄救場，而非系統
- 🚩 過度加班：加班成為常態，不是例外
- 🚩 信息封閉：重要決策不透明
- 🚩 對失敗零容忍：沒人敢嘗試新事物

## 5.6 領導力

### 5.6.1 技術領導 vs. 管理

| 角色 | 焦點 | 權力來源 |
|------|------|----------|
| 技術領導 | 技術方向、架構決策、技術成長 | 專業能力、影響力 |
| 管理 | 資源分配、進度控制、人員管理 | 職位權力 |

在軟體團隊中，兩者都需要，但不一定是同一個人。

### 5.6.2 僕人式領導（Servant Leadership）

敏捷強調的領導風格：

> 「領導者的角色是服務團隊，而非控制團隊。」

僕人式領導的行為：
- **移除障礙**：幫助團隊專注於有價值的工作
- **支持成長**：創造學習和成長的機會
- **傾聽**：理解團隊的需求和困難
- **授權**：信任團隊做決策
- **建立環境**：創造讓團隊成功的條件

### 5.6.3 情境領導

不同情境需要不同的領導風格：

| 成員成熟度 | 領導風格 | 行為 |
|------------|----------|------|
| 低能力、高意願 | 指導 | 告訴他們怎麼做 |
| 有些能力、低意願 | 教練 | 解釋為什麼，提供支持 |
| 高能力、意願不定 | 支持 | 參與決策，提供鼓勵 |
| 高能力、高意願 | 授權 | 放手讓他們做 |

**關鍵**：沒有「最佳」領導風格，要根據情境調整。

## 5.7 本章總結

### 核心概念

| 概念 | 一句話解釋 |
|------|-----------|
| 人因優先 | 軟體專案失敗的主因是人和溝通，不是技術 |
| 小團隊高效 | 溝通成本隨人數指數增長，小團隊效率最高 |
| 團隊發展階段 | 形成→風暴→規範→績效，風暴期是正常的 |
| 建設性衝突 | 衝突本身不是問題，如何處理才是問題 |
| 心理安全 | 高效能團隊最重要的特徵 |
| 僕人式領導 | 領導者服務團隊，而非控制團隊 |

### 實踐指引

1. **溝通**：確認理解、視覺化、選擇適當通道
2. **衝突**：對事不對人、尋求合作
3. **文化**：建立心理安全、允許失敗
4. **領導**：根據情境調整風格、服務團隊
