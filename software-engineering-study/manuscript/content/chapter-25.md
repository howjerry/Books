# 第25章　可行且有用的軟體計畫

> 「計畫的價值不在於預測未來，而在於準備好面對未來。」

---

## 25.1 為什麼需要計畫？

### 25.1.1 計畫的目的

| 目的 | 說明 |
|------|------|
| **協調** | 讓團隊朝同一方向前進 |
| **溝通** | 和利害關係人對齊期待 |
| **控制** | 提供追蹤進度的基準 |
| **風險** | 識別和準備風險 |
| **資源** | 規劃所需資源 |

### 25.1.2 計畫 vs. 規劃

| 計畫（Plan） | 規劃（Planning） |
|--------------|------------------|
| 文件 | 活動 |
| 靜態 | 動態 |
| 會過時 | 持續進行 |

**敏捷觀點**：
> 「Responding to change over following a plan.」
> 但這不是說不需要計畫，而是計畫要能適應變化。

### 25.1.3 好計畫的特徵

| 特徵 | 說明 |
|------|------|
| **可行** | 基於實際能力 |
| **可測** | 能衡量進度 |
| **可調** | 能適應變化 |
| **清楚** | 團隊能理解 |
| **完整** | 涵蓋必要面向 |

## 25.2 工作分解結構（WBS）

### 25.2.1 什麼是 WBS？

**WBS（Work Breakdown Structure）**：將專案工作分解成可管理的小單位。

```
┌──────────────────────────────────────────────────────────────┐
│                         專案                                  │
│                          │                                   │
│         ┌────────────────┼────────────────┐                  │
│         │                │                │                  │
│         ▼                ▼                ▼                  │
│    ┌─────────┐     ┌─────────┐     ┌─────────┐              │
│    │ 需求    │     │ 設計    │     │ 開發    │              │
│    └────┬────┘     └────┬────┘     └────┬────┘              │
│         │               │               │                    │
│    ┌────┴────┐     ┌────┴────┐     ┌────┴────┐              │
│    │         │     │         │     │         │              │
│    ▼         ▼     ▼         ▼     ▼         ▼              │
│  用戶    功能    架構    UI     模組A   模組B               │
│  訪談    規格    設計    設計    開發    開發               │
└──────────────────────────────────────────────────────────────┘
```

### 25.2.2 WBS 分解原則

| 原則 | 說明 |
|------|------|
| **100% 原則** | 子項加起來等於父項 |
| **獨立性** | 子項之間不重疊 |
| **可交付** | 最底層有明確交付物 |
| **可估算** | 最底層可估算時間 |

### 25.2.3 WBS 層級

| 層級 | 說明 | 範例 |
|------|------|------|
| Level 0 | 專案 | 電商系統 |
| Level 1 | 階段/模組 | 用戶管理、訂單管理 |
| Level 2 | 功能 | 用戶註冊、用戶登入 |
| Level 3 | 工作包 | 設計 UI、開發 API、測試 |

**建議**：分解到 8-80 小時的工作包。

## 25.3 估算

### 25.3.1 估算的困難

| 困難 | 原因 |
|------|------|
| **不確定性** | 軟體開發本質不確定 |
| **複雜性** | 難以預見所有情況 |
| **樂觀偏見** | 人傾向低估困難 |
| **壓力** | 被迫給出不實際的估計 |

### 25.3.2 估算方法

| 方法 | 說明 | 適用 |
|------|------|------|
| **專家判斷** | 基於經驗估算 | 有經驗的團隊 |
| **類比** | 參考類似專案 | 有歷史資料 |
| **分解** | 分解後加總 | 可分解的工作 |
| **三點估算** | 樂觀、悲觀、最可能 | 高不確定性 |
| **故事點** | 相對大小估算 | 敏捷團隊 |

### 25.3.3 三點估算

```
預期值 = (樂觀 + 4×最可能 + 悲觀) / 6
標準差 = (悲觀 - 樂觀) / 6
```

**範例**：

| 任務 | 樂觀 | 最可能 | 悲觀 | 預期 |
|------|------|--------|------|------|
| 用戶 API | 2天 | 3天 | 8天 | 3.7天 |
| 登入 UI | 1天 | 2天 | 5天 | 2.3天 |

### 25.3.4 故事點

| 點數 | 相對大小 | 範例 |
|------|----------|------|
| 1 | 很小 | 改一個文字 |
| 2 | 小 | 加一個欄位 |
| 3 | 中小 | 新增簡單頁面 |
| 5 | 中 | 新增功能模組 |
| 8 | 大 | 複雜功能 |
| 13 | 很大 | 需要拆分 |

**速度（Velocity）**：團隊每 Sprint 完成的故事點數。

## 25.4 時程規劃

### 25.4.1 建立時程的步驟

```
┌──────────────────────────────────────────────────────────────┐
│                    時程規劃步驟                               │
│                                                              │
│  1. 識別活動     2. 排序活動     3. 估算時間                 │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                  │
│  │ 列出所有 │───►│ 識別相依 │───►│ 估算每個 │                  │
│  │ 工作項目 │    │ 關係    │    │ 活動時間 │                  │
│  └─────────┘    └─────────┘    └─────────┘                  │
│                                      │                       │
│  ┌─────────┐    ┌─────────┐         │                       │
│  │ 識別關鍵│◄───│ 分配資源 │◄────────┘                       │
│  │ 路徑    │    │         │                                  │
│  └─────────┘    └─────────┘                                  │
└──────────────────────────────────────────────────────────────┘
```

### 25.4.2 活動相依性

| 類型 | 說明 | 範例 |
|------|------|------|
| **FS** | 前者完成後者開始 | 設計完成後開發 |
| **SS** | 同時開始 | 開發和測試計畫 |
| **FF** | 同時完成 | 開發和文件 |
| **SF** | 前者開始後者完成 | 很少用 |

### 25.4.3 甘特圖

```
任務              週1   週2   週3   週4   週5   週6
──────────────────────────────────────────────────
需求分析          ████
架構設計               ████
模組 A 開發                 ████████
模組 B 開發                 ████████
整合測試                              ████
用戶測試                                   ████
──────────────────────────────────────────────────
里程碑                  ◆               ◆    ◆
                     設計完成        開發完成  上線
```

### 25.4.4 關鍵路徑

**關鍵路徑（Critical Path）**：決定專案最短時間的最長路徑。

- 關鍵路徑上的任務沒有浮動時間
- 延遲任何一個都會延遲專案
- 是管理重點

## 25.5 敏捷規劃

### 25.5.1 規劃層次

```
┌──────────────────────────────────────────────────────────────┐
│                    敏捷規劃洋蔥                               │
│                                                              │
│                    ┌───────────┐                            │
│                   ╱   願景規劃  ╲                           │
│                  ╱   (1-3 年)    ╲                          │
│                 ╱─────────────────╲                         │
│                ╱                   ╲                        │
│               ╱     路線圖規劃      ╲                       │
│              ╱     (3-12 個月)      ╲                       │
│             ╱───────────────────────╲                      │
│            ╱                         ╲                     │
│           ╱       發布規劃            ╲                    │
│          ╱       (1-3 個月)           ╲                    │
│         ╱─────────────────────────────╲                   │
│        ╱                               ╲                  │
│       ╱         Sprint 規劃             ╲                 │
│      ╱          (1-4 週)                ╲                 │
│     ╱─────────────────────────────────────╲               │
│    ╱                                       ╲              │
│   ╱            每日規劃                     ╲             │
│  ╱             (每天)                        ╲            │
│ ╱─────────────────────────────────────────────╲          │
└──────────────────────────────────────────────────────────────┘
```

### 25.5.2 發布規劃

| 要素 | 說明 |
|------|------|
| **目標** | 這個發布要達成什麼？ |
| **範圍** | 包含哪些功能？ |
| **時間** | 預計何時發布？ |
| **速度** | 團隊的歷史速度？ |

**計算**：
```
發布範圍（故事點）÷ 團隊速度 = 需要的 Sprint 數
```

### 25.5.3 Sprint 規劃

| 步驟 | 活動 |
|------|------|
| 1 | 選擇 Sprint 目標 |
| 2 | 從 Backlog 選擇故事 |
| 3 | 分解成任務 |
| 4 | 確認承諾 |

## 25.6 緩衝管理

### 25.6.1 為什麼需要緩衝？

| 原因 | 說明 |
|------|------|
| **估算不準** | 估算必然有誤差 |
| **未知風險** | 無法預見的問題 |
| **外部依賴** | 等待他人 |
| **學生症候群** | 拖到最後一刻 |

### 25.6.2 緩衝類型

| 類型 | 說明 |
|------|------|
| **專案緩衝** | 放在專案最後 |
| **接駁緩衝** | 放在非關鍵路徑和關鍵路徑交接處 |
| **資源緩衝** | 關鍵資源開始前提醒 |

### 25.6.3 緩衝大小

經驗法則：
- 低風險：10-15%
- 中風險：15-25%
- 高風險：25-50%

| 概念 | 一句話解釋 |
|------|-----------|
