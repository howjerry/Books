## 30.2 軟體工程的不變原則

### 30.2.1 技術會變，原則不變

```
┌──────────────────────────────────────────────────────────────┐
│                    不變的核心原則                             │
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    簡單                              │   │
│   │    KISS - 保持簡單，避免過度複雜                      │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    分解                              │   │
│   │    把大問題分解成小問題                               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    抽象                              │   │
│   │    隱藏複雜性，暴露簡單介面                           │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                    反饋                              │   │
│   │    快速反饋，持續改善                                 │   │
│   └─────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 30.2.2 十大永恆原則

| # | 原則 | 說明 |
|---|------|------|
| 1 | **分離關注點** | 每個模組做一件事 |
| 2 | **單一責任** | 變化的原因只有一個 |
| 3 | **開放封閉** | 對擴展開放，對修改封閉 |
| 4 | **依賴倒置** | 依賴抽象，不依賴具體 |
| 5 | **高內聚低耦合** | 模組內緊密，模組間鬆散 |
| 6 | **DRY** | 不要重複自己 |
| 7 | **KISS** | 保持簡單 |
| 8 | **YAGNI** | 不要預做不需要的 |
| 9 | **快速反饋** | 盡早發現問題 |
| 10 | **持續改善** | 永遠可以更好 |

### 30.2.3 這些原則為什麼不變？

因為它們解決的是軟體的根本挑戰：

| 挑戰 | 對應原則 |
|------|----------|
| **複雜性** | 分解、抽象、簡單 |
| **變化** | 開放封閉、鬆耦合 |
| **溝通** | 單一責任、清晰介面 |
| **品質** | 快速反饋、持續改善 |

## 30.5 軟體工程的未來

### 30.5.1 不變的挑戰

| 挑戰 | 為什麼不變 |
|------|-----------|
| **複雜性** | 需求越來越複雜 |
| **變化** | 業務永遠在變 |
| **溝通** | 人與人的協作 |
| **品質** | 永遠追求更好 |

### 30.5.2 變化的工具

| 過去 | 現在 | 未來 |
|------|------|------|
| 手工編碼 | IDE 輔助 | AI 生成 |
| 手動部署 | CI/CD | 自動化平台 |
| 手動測試 | 自動化測試 | AI 測試 |
| 文件查閱 | 搜尋引擎 | AI 助手 |

### 30.5.3 軟體工程師的價值

即使 AI 越來越強大，軟體工程師仍然需要：

| 價值 | 說明 |
|------|------|
| **問題定義** | AI 需要知道解決什麼 |
| **系統設計** | 大局觀和架構能力 |
| **判斷決策** | 權衡和選擇 |
| **品質把關** | 審查和驗證 |
| **持續學習** | 適應變化 |

## 30.6 最後的話

### 30.6.1 軟體工程是一門藝術

軟體工程既是科學也是藝術：

| 科學面 | 藝術面 |
|--------|--------|
| 有方法論 | 需要創意 |
| 可度量 | 有美感 |
| 有規則 | 有風格 |
| 可學習 | 需要悟性 |

### 30.6.3 永遠記住

> **「好的軟體是為人服務的。技術是手段，不是目的。」**

- 使用者需要能解決問題的軟體
- 團隊成員需要能維護的軟體
- 業務需要能創造價值的軟體

這才是軟體工程的終極目標。
