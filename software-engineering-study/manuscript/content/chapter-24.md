# 第24章　專案管理的概念

> 「專案管理的本質是在不確定性下做承諾，並持續調整承諾。」

---

## 24.1 什麼是軟體專案管理？

### 24.1.1 專案的定義

**專案（Project）**：有明確開始和結束，產出獨特產品或服務的臨時性工作。

| 專案 | 營運 |
|------|------|
| 臨時的 | 持續的 |
| 獨特的 | 重複的 |
| 漸進明細 | 穩定流程 |

### 24.1.2 軟體專案的特殊性

| 特性 | 挑戰 |
|------|------|
| **無形** | 難以看到進度 |
| **複雜** | 難以估計 |
| **可變** | 需求經常變動 |
| **知識密集** | 依賴人的能力 |
| **技術風險** | 新技術不確定性 |

### 24.1.3 專案管理的四個限制

```
                    範圍
                     │
                     │
         ┌───────────┼───────────┐
         │           │           │
         │     ┌─────┴─────┐     │
         │     │           │     │
    時間 ────► │   品質    │◄──── 成本
         │     │           │     │
         │     └───────────┘     │
         │                       │
         └───────────────────────┘
```

**鐵三角**：範圍、時間、成本——改變一個會影響其他。

**現實**：
- 固定時間 + 固定範圍 = 品質下降
- 固定品質 + 固定範圍 = 時間增加
- 固定時間 + 固定品質 = 範圍縮減

## 24.2 專案經理的角色

### 24.2.1 專案經理的職責

| 職責 | 說明 |
|------|------|
| **規劃** | 定義範圍、時程、資源 |
| **組織** | 建立團隊、分配角色 |
| **領導** | 激勵、協調、決策 |
| **控制** | 監控進度、管理變更 |
| **溝通** | 和利害關係人溝通 |

### 24.2.2 專案經理的技能

```

│                    專案經理技能金字塔                         │

│                 ╱                   ╲                        │
│                ╱     專案管理技術    ╲                       │
│               ╱          ●          ╲                        │
│              ╱───────────────────────╲                       │
│             ╱                         ╲                      │
```

| 技能類別 | 具體技能 |
|----------|----------|
| **技術能力** | 了解軟體開發、工具、技術 |
| **專案管理** | 規劃、估算、追蹤、風險管理 |
| **溝通** | 報告、談判、衝突處理 |
| **領導力** | 願景、激勵、決策、影響力 |

### 24.2.3 軟體 PM vs. 傳統 PM

| 面向 | 傳統 PM | 軟體 PM |
|------|---------|---------|
| 產品 | 有形 | 無形 |
| 需求 | 較穩定 | 經常變動 |
| 進度 | 可見 | 難以衡量 |
| 估算 | 較準確 | 高度不確定 |
| 團隊 | 可替換 | 高度依賴個人 |

## 24.3 專案生命週期

### 24.3.1 專案階段

```
┌──────────────────────────────────────────────────────────────┐
│                     專案生命週期                              │
│                                                              │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐ │
│  │  啟動  │─►│  規劃  │─►│  執行  │─►│ 監控   │─►│  結案  │ │
│  │        │  │        │  │        │  │        │  │        │ │
│  └────────┘  └────────┘  └────────┘  └────────┘  └────────┘ │
│                                                              │
│      │           │           │           │           │       │
│      ▼           ▼           ▼           ▼           ▼       │
│   專案章程    專案計畫    交付物      狀態報告    經驗教訓    │
│   利害關係人  WBS        進度更新    變更請求    結案報告    │
│   目標定義    時程       風險處理    問題追蹤    資產移交    │
└──────────────────────────────────────────────────────────────┘
```

### 24.3.2 各階段活動

| 階段 | 主要活動 | 產出 |
|------|----------|------|
| **啟動** | 定義目標、識別利害關係人 | 專案章程 |
| **規劃** | 分解工作、估算、排程 | 專案計畫 |
| **執行** | 開發、測試、整合 | 交付物 |
| **監控** | 追蹤進度、管理變更 | 狀態報告 |
| **結案** | 驗收、經驗學習 | 結案報告 |

### 24.3.3 敏捷 vs. 預測式生命週期

| 面向 | 預測式（Waterfall） | 敏捷（Agile） |
|------|---------------------|---------------|
| 規劃 | 前期完整規劃 | 漸進式規劃 |
| 需求 | 固定 | 可變 |
| 交付 | 最後交付 | 頻繁交付 |
| 變更 | 抗拒變更 | 擁抱變更 |
| 回饋 | 晚期 | 持續 |

## 24.4 利害關係人管理

### 24.4.1 識別利害關係人

| 利害關係人 | 關注點 |
|------------|--------|
| **客戶/用戶** | 功能、品質、交期 |
| **高層主管** | ROI、策略對齊 |
| **團隊成員** | 工作環境、成長 |
| **運維團隊** | 可維護性、部署 |
| **供應商** | 合約、付款 |

### 24.4.2 利害關係人矩陣

```
                    影響力
              低            高
         ┌─────────┬─────────┐
      高 │  保持   │  管理   │
   利    │  滿意   │  密切   │
   益    ├─────────┼─────────┤
      低 │  監控   │  保持   │
         │  即可   │  知情   │
         └─────────┴─────────┘
```

| 象限 | 策略 |
|------|------|
| 高利益 + 高影響 | 密切管理，頻繁溝通 |
| 高利益 + 低影響 | 保持滿意，定期更新 |
| 低利益 + 高影響 | 保持知情，關鍵時刻溝通 |
| 低利益 + 低影響 | 監控，必要時溝通 |

### 24.4.3 溝通計畫

| 對象 | 內容 | 頻率 | 方式 |
|------|------|------|------|
| 高層主管 | 進度摘要、風險、決策 | 雙週 | 報告、會議 |
| 客戶 | 進度、Demo | 每週 | 會議 |
| 團隊 | 日常、阻礙 | 每日 | 站會 |
| 利害關係人 | 里程碑 | 里程碑時 | 郵件 |

## 24.5 團隊管理

### 24.5.1 團隊發展階段

**Tuckman 模型**：

```
┌──────────────────────────────────────────────────────────────┐
│                   團隊發展階段                                │
│                                                              │
│   績效                                                       │
│    ▲                                        ┌───────────┐   │
│    │                              ┌─────────┤ Performing │   │
│    │                    ┌─────────┤         │ (高績效)   │   │
│    │          ┌─────────┤ Norming │         └───────────┘   │
│    │ Forming  │ Storming│ (規範)  │                         │
│    │ (形成)   │ (風暴)  │         │                         │
│    └──────────┴─────────┴─────────┴─────────────────► 時間  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

| 階段 | 特徵 | PM 角色 |
|------|------|---------|
| **Forming** | 禮貌、觀望 | 指導、建立規則 |
| **Storming** | 衝突、競爭 | 調解、促進討論 |
| **Norming** | 合作、建立規範 | 支持、授權 |
| **Performing** | 高效、自主 | 授權、移除障礙 |

### 24.5.2 團隊規模

| 規模 | 說明 |
|------|------|
| 2 人 | 溝通路徑 1 條 |
| 5 人 | 溝通路徑 10 條 |
| 10 人 | 溝通路徑 45 條 |
| 20 人 | 溝通路徑 190 條 |

**公式**：n(n-1)/2

**建議**：
- 核心團隊 5-9 人
- 超過則拆分子團隊

### 24.5.3 動機理論

**Herzberg 雙因素理論**：

| 保健因素（不滿足會不滿） | 激勵因素（滿足會滿意） |
|--------------------------|------------------------|
| 薪資 | 成就感 |
| 工作環境 | 認可 |
| 公司政策 | 工作本身 |
| 人際關係 | 責任 |
| 工作安全 | 成長 |

## 24.6 本章總結

### 核心概念

| 概念 | 一句話解釋 |
|------|-----------|
| 專案 | 有開始有結束的臨時性工作 |
| 鐵三角 | 範圍、時間、成本相互制約 |
| 利害關係人 | 影響或被影響的人 |
| 團隊發展 | Forming → Storming → Norming → Performing |

### PM 核心職責 Checklist

| 職責 | 問題 |
|------|------|
| 範圍 | 清楚定義了嗎？ |
| 時程 | 可行的時程嗎？ |
| 資源 | 資源足夠嗎？ |
| 風險 | 識別並管理風險嗎？ |
| 溝通 | 和利害關係人溝通嗎？ |
| 品質 | 品質標準定義了嗎？ |
