# 第3章　敏捷與流程

> 「敏捷不是沒有流程，而是用最少的流程達成最大的價值。」

---

## 3.1 敏捷的起源：一場反抗運動

### 2001 年的雪鳥會議

2001 年 2 月，17 位軟體開發者在美國猶他州的雪鳥滑雪場聚會。他們來自不同的陣營——XP、Scrum、DSDM、Crystal——但有一個共同的敵人：**重量級流程**。

那個年代，軟體開發被大量的文件、審批、計畫所淹沒。「流程」成了目的本身，而非達成目的的手段。

這 17 個人決定找出他們的共同信念，結果就是著名的《敏捷軟體開發宣言》。

### 敏捷宣言

> **我們正在透過實踐和幫助他人實踐，發掘更好的軟體開發方法。透過這項工作，我們得到以下價值觀：**
>
> - **個人與互動** 重於 流程與工具
> - **可用的軟體** 重於 詳盡的文件
> - **與客戶合作** 重於 合約協商
> - **回應變化** 重於 遵循計畫
>
> **也就是說，雖然右邊的項目有價值，但我們更重視左邊的項目。**

### 被誤解的宣言

這份宣言是軟體史上被誤解最多的文件之一。

**常見誤解**：
- ❌ 敏捷 = 不需要流程
- ❌ 敏捷 = 不需要文件
- ❌ 敏捷 = 不需要計畫
- ❌ 敏捷 = 可以隨意變更

**正確理解**：
- ✅ 敏捷 = 流程服務於人，而非人服務於流程
- ✅ 敏捷 = 只寫必要的文件，但必要的要寫
- ✅ 敏捷 = 有計畫，但計畫可以調整
- ✅ 敏捷 = 擁抱變化，但有管理地變化

**關鍵詞是「重於」而非「取代」**。右邊的項目仍有價值，只是左邊更重要。

## 3.2 敏捷的十二原則

宣言背後有十二條原則，這些原則比宣言本身更具體，更有指導意義。

### 原則一覽

| # | 原則 | 核心含義 |
|---|------|----------|
| 1 | 最高優先級是透過**早期和持續交付**有價值的軟體來滿足客戶 | 交付 > 承諾 |
| 2 | 歡迎變更需求，即使在開發後期 | 變化是常態 |
| 3 | 頻繁交付可用的軟體，週期越短越好 | 小批量、快回饋 |
| 4 | 業務人員和開發人員必須**每天一起工作** | 緊密協作 |
| 5 | 以有動力的個人為核心建立專案，給予環境和支持，信任他們完成工作 | 信任與授權 |
| 6 | 面對面交談是最有效的溝通方式 | 高頻寬溝通 |
| 7 | **可用的軟體**是進度的主要度量標準 | 不是文件，是軟體 |
| 8 | 敏捷流程提倡可持續的開發步調 | 不是衝刺是馬拉松 |
| 9 | 持續關注卓越的技術和良好的設計 | 技術卓越 |
| 10 | 簡單——**最大化未完成工作量**的藝術——是必要的 | 少即是多 |
| 11 | 最佳的架構、需求和設計來自**自組織團隊** | 自主管理 |
| 12 | 團隊定期反思如何更有效，然後相應地調整行為 | 持續改進 |

### 三個最重要的原則

如果只能記住三個，我會選擇：

**原則 3：頻繁交付**
> 交付週期越短，回饋越快，問題越早發現，變更成本越低。

**原則 7：可用的軟體是進度的度量**
> 不是計畫完成度、不是文件完成度、不是會議次數——是可用的軟體。

**原則 12：定期反思並調整**
> 沒有完美的流程，只有持續改進的流程。

## 3.3 敏捷的本質：快速回饋迴路

所有敏捷實踐背後有一個共同的原理：**縮短回饋迴路**。

### 什麼是回饋迴路？

```
    做出假設 ──────────► 採取行動
        ▲                    │
        │                    │
        │                    ▼
     調整假設 ◄────────── 觀察結果
```

回饋迴路越短，你就能越快知道：
- 做對了還是做錯了
- 假設成立還是不成立
- 該繼續還是該調整

### 各層級的回饋迴路

| 層級 | 回饋來源 | 週期 | 敏捷實踐 |
|------|----------|------|----------|
| 程式碼 | 編譯器、測試 | 秒~分鐘 | TDD、持續整合 |
| 功能 | 產品負責人 | 天~週 | Sprint Review、Demo |
| 產品 | 真實使用者 | 週~月 | 使用者測試、A/B 測試 |
| 方向 | 市場 | 月~季 | MVP、精實創業 |

### 敏捷 vs. 瀑布的本質差異

瀑布的問題不是「有流程」，而是**回饋太晚**。

```
瀑布：
需求 ─────────────────────────────────────────► 測試 ─► 回饋
      （6-12 個月後才知道做對了沒）

敏捷：
需求 ─► 開發 ─► 回饋 ─► 調整 ─► 開發 ─► 回饋 ─► ...
      （2-4 週就知道做對了沒）
```

## 3.4 主流敏捷方法

### 3.4.1 Scrum

**起源**：Jeff Sutherland 和 Ken Schwaber，1990 年代

**核心元素**：

| 元素 | 說明 |
|------|------|
| **Sprint** | 固定時間盒（2-4 週），結束時交付可用的增量 |
| **Product Backlog** | 待辦事項清單，按優先級排序 |
| **Sprint Backlog** | 本 Sprint 承諾完成的事項 |
| **Daily Scrum** | 每日站會，15 分鐘，同步進度和障礙 |
| **Sprint Review** | Sprint 結束時展示成果，收集回饋 |
| **Sprint Retrospective** | Sprint 結束時反思，決定改進事項 |

**三個角色**：

| 角色 | 責任 |
|------|------|
| **Product Owner** | 定義需求、排優先級、接受交付 |
| **Scrum Master** | 保護團隊、移除障礙、確保流程 |
| **Development Team** | 交付可用的軟體（跨功能、自組織） |

**適用情境**：
- ✅ 需求會變化，需要定期調整優先級
- ✅ 可以取得使用者/產品負責人的定期回饋
- ✅ 團隊相對穩定，可以建立節奏
- ❌ 緊急維護、隨叫隨到的支援工作

### 3.4.2 看板（Kanban）

**起源**：源自豐田生產系統，David Anderson 應用於軟體開發

**核心原則**：

1. **視覺化工作流程**：將工作變成看得見的卡片
2. **限制在製品（WIP）**：每個階段限制同時進行的工作數量
3. **管理流動**：關注工作的流動，而非個人的忙碌
4. **明確政策**：讓「完成」的定義明確
5. **實施回饋迴路**：定期審視和改進
6. **協作改進，實驗演進**：漸進式改變

**看板圖**：

```
待辦     │  分析 (2)  │  開發 (3)  │  測試 (2)  │  完成
─────────┼────────────┼────────────┼────────────┼─────────
[任務1]  │  [任務4]   │  [任務6]   │  [任務8]   │ [任務10]
[任務2]  │  [任務5]   │  [任務7]   │  [任務9]   │ [任務11]
[任務3]  │            │  [任務X]   │            │
         │            │   ↑ WIP=3 │            │
         │            │  不能再加  │            │
```

**適用情境**：
- ✅ 工作項目大小差異大
- ✅ 需要處理緊急插件
- ✅ 現有流程想漸進改善
- ❌ 需要預測何時完成特定功能

### 3.4.3 極限編程（XP）

**起源**：Kent Beck，1990 年代

**核心實踐**：

| 實踐 | 說明 |
|------|------|
| **結對編程** | 兩人一起寫程式，即時審查 |
| **測試驅動開發（TDD）** | 先寫測試，再寫程式碼 |
| **持續整合** | 頻繁整合程式碼，自動測試 |
| **重構** | 持續改善程式碼結構 |
| **簡單設計** | 只做當前需要的 |
| **集體所有權** | 任何人可以改任何程式碼 |
| **持續部署** | 頻繁部署到生產環境 |
| **現場客戶** | 客戶隨時可以提問 |

**適用情境**：
- ✅ 團隊願意接受嚴格的技術實踐
- ✅ 程式碼品質是首要考量
- ✅ 需求變化快，需要快速回饋
- ❌ 團隊分散難以結對
- ❌ 遺留系統難以建立測試

### 3.4.4 比較

| 方法 | 焦點 | 節奏 | 特色 |
|------|------|------|------|
| Scrum | 團隊協作 | 固定 Sprint | 角色、儀式、產物 |
| Kanban | 流動效率 | 連續流動 | WIP 限制、視覺化 |
| XP | 技術卓越 | 極短迭代 | 工程實踐 |

## 3.5 敏捷的陷阱

敏捷不是銀彈。許多團隊「導入敏捷」後發現問題依舊，有時甚至更糟。

### 陷阱 1：名義敏捷（Agile in Name Only）

**症狀**：
- 有 Sprint，但沒有可用的增量
- 有 Daily Standup，但只是報告進度
- 有 Retrospective，但從不改變任何事

**根因**：只學了形式，沒有理解精神

**對策**：回到原則——可用的軟體、回饋、持續改進

### 陷阱 2：無紀律的敏捷（Fragile Agile）

**症狀**：
- 「敏捷」成了不寫文件的藉口
- 「擁抱變化」成了不規劃的藉口
- 技術債堆積，越來越難交付

**根因**：敏捷需要更高的紀律，不是更低

**對策**：堅持技術實踐（測試、重構、持續整合）

### 陷阱 3：敏捷教條主義（Agile Fundamentalism）

**症狀**：
- 「Scrum Guide 說不能這樣」
- 拒絕任何調整，即使不適合
- 形式比結果重要

**根因**：忘了敏捷的核心是「適應」

**對策**：敏捷本身也要敏捷——根據情境調整

### 陷阱 4：假敏捷（Dark Agile）

**症狀**：
- 用「敏捷」壓縮時程
- 用「擁抱變化」頻繁變更範圍但不調整時程
- 用「自組織」推卸管理責任

**根因**：誤用敏捷來掩蓋管理問題

**對策**：敏捷不是管理者的工具，是團隊的工具

## 3.6 敏捷的適用情境

敏捷不是萬能的。它有明確的適用範圍。

### 適合敏捷的情境

| 情境 | 原因 |
|------|------|
| 需求不明確或會變化 | 敏捷擁抱變化 |
| 可以獲得頻繁的使用者回饋 | 回饋是敏捷的燃料 |
| 團隊共處或溝通順暢 | 敏捷依賴高頻寬溝通 |
| 可以頻繁交付 | 部署管道暢通 |
| 團隊有自組織的能力和意願 | 敏捷依賴團隊自主 |

### 不適合敏捷的情境

| 情境 | 原因 |
|------|------|
| 需求穩定、合約固定 | 敏捷的優勢無法發揮 |
| 無法獲得使用者回饋 | 敏捷失去了回饋迴路 |
| 嚴格的合規/文件要求 | 敏捷的輕量級可能不夠 |
| 團隊分散、溝通成本高 | 敏捷的溝通假設不成立 |
| 團隊不穩定、能力不足 | 自組織需要基礎能力 |

### 務實的選擇

與其問「該不該用敏捷」，不如問：

1. **我能多快獲得回饋？** → 越快，越適合敏捷
2. **變更的可能性多高？** → 越高，越適合敏捷
3. **團隊能自組織嗎？** → 能，則敏捷效果更好
4. **有哪些限制必須遵守？** → 據此調整敏捷實踐

## 3.7 本章總結

### 核心概念

| 概念 | 一句話解釋 |
|------|-----------|
| 敏捷宣言 | 價值觀的表達，「重於」不是「取代」 |
| 敏捷原則 | 比宣言更具體的指導方針 |
| 回饋迴路 | 敏捷的本質是縮短回饋迴路 |
| Scrum | 以 Sprint 為節奏的團隊協作框架 |
| Kanban | 限制 WIP、管理流動的方法 |
| XP | 強調技術卓越的工程實踐 |

### 關鍵決策

採用敏捷前問自己：
1. 我能獲得真正的使用者回饋嗎？
2. 團隊有自組織的能力嗎？
3. 我願意承擔敏捷需要的紀律嗎？
4. 我的限制條件允許敏捷嗎？
