### 我現在的理解是什麼？

閱讀前，我的理解：
- 估算 = 猜
- 計畫總是不準
- 加班可以解決延遲

### 我遇過哪些相關痛點？

1. **估算不準**：說兩天，做了一週
2. **沒有緩衝**：一點延遲就整個趕
3. **計畫沒用**：做了計畫沒人看

## 本章一句話

> **好的計畫是可行、可測、可調的。使用 WBS 分解工作，用三點估算處理不確定性，加入適當緩衝。計畫是持續的活動，不是一次性的文件。**

## 3 個關鍵概念

### 概念 1：WBS 分解

**書中定義**：

將大工作分解成小的可管理單位。

| 原則 | 說明 |
|------|------|
| 100% 原則 | 子項加起來 = 父項 |
| 獨立性 | 子項不重疊 |
| 可交付 | 最底層有明確產出 |
| 可估算 | 8-80 小時 |

**我的理解**：

分解是估算的基礎。分得越細，估算越準。

**範例**：

```
用戶登入功能
├── 需求（8h）
│   ├── 用戶訪談（4h）
│   └── 規格撰寫（4h）
├── 設計（16h）
│   ├── UI 設計（8h）
│   └── API 設計（8h）
├── 開發（40h）
│   ├── 前端（16h）
│   ├── 後端（16h）
│   └── 整合（8h）
└── 測試（16h）
    ├── 單元測試（8h）
    └── 整合測試（8h）

總計：80h
```

**經驗法則**：
> 「如果說不出怎麼分解，代表還沒想清楚。」

---

### 概念 2：三點估算

**書中定義**：

```
預期值 = (樂觀 + 4×最可能 + 悲觀) / 6
```

**我的理解**：

這解決了「單點估算太武斷」的問題。

**使用方法**：

| 問題 | 估算 |
|------|------|
| 「最快多久？」 | 樂觀 = 2天 |
| 「正常多久？」 | 最可能 = 3天 |
| 「最慢多久？」 | 悲觀 = 8天 |

預期 = (2 + 4×3 + 8) / 6 = 3.7 天

**我的實踐**：

以前：「這個 3 天」
現在：「這個最快 2 天，正常 3 天，最慢可能 5 天」

**經驗法則**：
> 「單點估算是假裝確定，三點估算是承認不確定。」

---

### 概念 3：緩衝管理

**書中定義**：

在計畫中加入緩衝時間，應對不確定性。

| 風險等級 | 緩衝比例 |
|----------|----------|
| 低 | 10-15% |
| 中 | 15-25% |
| 高 | 25-50% |

**我的理解**：

緩衝不是「偷懶的時間」，是「應對意外的保險」。

**沒有緩衝的問題**：

- 一點延遲就整個趕
- 團隊持續高壓
- 品質下降
- 技術債累積

**緩衝的使用原則**：

1. **集中管理**：緩衝歸專案經理管
2. **不告訴團隊具體數字**：避免「學生症候群」
3. **追蹤消耗**：監控緩衝使用情況

**經驗法則**：
> 「沒有緩衝的計畫是在賭博。」

## 原則、方法、護欄

### 估算方法速查表

| 情境 | 建議方法 |
|------|----------|
| 有類似專案經驗 | 類比估算 |
| 可以分解工作 | 分解估算 |
| 不確定性高 | 三點估算 |
| 敏捷團隊 | 故事點 + 速度 |
| 完全未知 | 先做 Spike 再估 |

---

### 護欄（Guardrail）

| 危險信號 | 可能的問題 | 修正方式 |
|----------|-----------|----------|
| 沒有分解就估算 | 估算不準 | 先分解再估 |
| 只有單點估算 | 低估不確定性 | 用三點估算 |
| 沒有緩衝 | 一延遲就趕 | 加入緩衝 |
| 計畫從不更新 | 計畫和現實脫節 | 定期更新 |
| 忽略相依性 | 排程衝突 | 識別相依 |

## 1 個決策點

### 決策情境

**估算被要求壓縮，怎麼辦？**

### 選項對比

| 選項 | 做法 | 風險 |
|------|------|------|
| 接受壓縮 | 按要求的時間做 | 品質下降、超時 |
| 堅持估算 | 解釋為什麼需要這麼久 | 可能被否決 |
| 調整範圍 | 提出範圍縮減的選項 | 功能減少 |
| 增加資源 | 請求更多人力 | 成本增加 |

### 決策準則

| 情境 | 建議 |
|------|------|
| 有談判空間 | 解釋估算依據 |
| 時間硬限制 | 提出範圍調整 |
| 估算不確定 | 提出先做 Spike |
| 被迫接受 | 記錄風險、保留證據 |

**經驗法則**：
> 「壓縮時間可以，但要說清楚代價。」

**正確做法**：
- 「可以縮短到 X，但需要縮減 A 功能」
- 「如果時間不變，需要先做一週 Spike 來降低風險」

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「學生症候群」估算**

### 錯誤做法

因為有緩衝就不急：
- 「還有兩週，不急」
- 「還有時間，先做其他的」
- 「最後趕一趕就好」

結果：工作都擠到最後，緩衝用光還是趕。

### 後果

1. **緩衝無效**：有緩衝也還是趕
2. **品質下降**：最後趕工
3. **壓力集中**：最後幾天高壓

### 正確做法

1. **早點開始**
   - 「開始時間到了就開始」
   - 不要因為有緩衝就拖延

2. **追蹤進度**
   - 每天/每週追蹤
   - 早發現延遲

3. **緩衝集中管理**
   - 不要把緩衝分配到每個任務
   - 專案經理統一管理

**檢驗標準**：
> 「進度是線性的嗎？還是最後才趕？」

## 延伸思考

### 敏捷團隊還需要這些估算嗎？

**我的看法**：

需要，但形式不同。

| 傳統 | 敏捷 |
|------|------|
| 預估小時 | 故事點 |
| 甘特圖 | Burndown |
| 專案緩衝 | 速度穩定 |
| 事前完整計畫 | 漸進式規劃 |

**敏捷的估算優勢**：
- 用速度代替猜測
- 頻繁交付，頻繁校正
- 不假裝知道不知道的事

### 估算總是不準怎麼辦？

**我的策略**：

1. **接受不準是正常的**
   - 估算是預測，不是承諾
   - 記錄估算和實際的差距

2. **持續校正**
   - 分析為什麼不準
   - 調整估算方法

3. **建立歷史資料**
   - 類似功能花多久
   - 團隊速度是多少

4. **承認不知道**
   - 真的不知道就說不知道
   - 用 Spike 降低不確定性
