## 章前預測（5 分鐘）

### 這章應該回答什麼問題？

1. 什麼是 UX？和 UI 有什麼不同？
2. UX 設計有哪些原則？
3. 如何進行使用者研究和測試？
4. 什麼是可及性？為什麼重要？

### 我現在的理解是什麼？

閱讀前，我的理解：
- UX 就是讓介面好看
- UX 是設計師的事，和工程師無關
- 可及性是法規要求，不太重要

### 我遇過哪些相關痛點？

1. **使用者不會用**：功能明明在那，使用者就是找不到
2. **開發完才發現問題**：上線後使用者抱怨
3. **被忽略的可及性**：沒考慮鍵盤操作

## 本章一句話

> **UX 不是讓介面漂亮，而是讓使用者能順利完成任務。關鍵是降低認知負擔、保持一致性、提供清楚回饋。**

---

## 3 個關鍵概念

### 概念 1：認知負擔

**書中定義**：

認知負擔 = 使用者大腦處理資訊的負擔。

降低認知負擔的方法：
- 分塊：資訊分成小塊
- 漸進式揭露：只顯示當前需要的
- 預設值：提供合理預設
- 一致性：減少學習成本

**我的理解**：

這解釋了為什麼有些介面「看起來」簡單但「用起來」複雜。

**例子**：

```
❌ 高認知負擔
一個頁面 50 個欄位，全部一次顯示

✅ 低認知負擔
分成 5 步，每步 10 個欄位
進度條顯示「第 2 步，共 5 步」
```

**經驗法則**：
- 一次不超過 7±2 個項目（Miller's Law）
- 隱藏不常用的選項
- 提供合理預設

---

### 概念 2：尼爾森的 10 個可用性原則

**書中定義**：

最實用的前 5 個：
1. 系統狀態可見
2. 符合真實世界
3. 使用者控制（取消、返回）
4. 一致性
5. 預防錯誤

**我的理解**：

這些原則可以當作 **UX 審查清單**。

**應用到程式碼**：

| 原則 | 程式碼實踐 |
|------|-----------|
| 系統狀態可見 | Loading 狀態、錯誤訊息 |
| 使用者控制 | 確認對話框、復原功能 |
| 一致性 | 共用元件、設計系統 |
| 預防錯誤 | 表單驗證、禁用無效按鈕 |
| 錯誤處理 | 清楚的錯誤訊息 + 解法 |

**程式碼例子**：

```javascript
// ❌ 不好：沒有狀態回饋
async function submitForm() {
  await api.submit(data);
}

// ✅ 好：有狀態回饋
async function submitForm() {
  setLoading(true);
  try {
    await api.submit(data);
    showSuccess("儲存成功！");
  } catch (error) {
    showError("儲存失敗，請稍後再試");
  } finally {
    setLoading(false);
  }
}
```

### 概念 3：可及性（Accessibility）

**書中定義**：

讓所有人都能使用產品，包括身心障礙者。

WCAG 四原則：可感知、可操作、可理解、穩健。

**我的理解**：

可及性不只是「好人好事」，也是：
- **法規要求**：很多國家有法規
- **SEO 優化**：語意化有助搜尋
- **提升品質**：對所有人都更好用

**最低限度實踐**：

| 實踐 | 做法 | 工程師可以做的 |
|------|------|---------------|
| 替代文字 | 圖片加 alt | `<img alt="描述">` |
| 鍵盤支援 | 可用 Tab 操作 | 檢查 focus 順序 |
| 對比度 | 文字對比度 ≥ 4.5:1 | 用工具檢查 |
| 語意標籤 | 用正確的 HTML | `<button>` 而非 `<div onClick>` |
| 標籤關聯 | 表單標籤關聯 | `<label for="id">` |

**檢查工具**：
- Chrome DevTools Lighthouse
- axe DevTools
- WAVE

## 原則、方法、護欄

### UX 設計原則速查表

| 原則 | 問題 | 解法 |
|------|------|------|
| 系統狀態可見 | 使用者知道發生什麼嗎？ | Loading、進度條、訊息 |
| 一致性 | 同樣的東西呈現一樣嗎？ | 設計系統、共用元件 |
| 預防錯誤 | 如何防止錯誤？ | 驗證、禁用無效選項 |
| 認知負擔 | 資訊太多嗎？ | 分步驟、漸進式揭露 |
| 可及性 | 鍵盤可操作嗎？ | 語意標籤、對比度 |

---

### 護欄（Guardrail）

| 危險信號 | 可能的問題 | 修正方式 |
|----------|-----------|----------|
| 使用者找不到功能 | 資訊架構不良 | 使用者測試、優化導航 |
| 操作後沒回饋 | 系統狀態不可見 | 加 Loading、成功/錯誤訊息 |
| 使用者常犯錯 | 缺少預防機制 | 驗證、確認對話框 |
| 無法鍵盤操作 | 可及性問題 | 檢查 Tab 順序、語意標籤 |
| 錯誤訊息看不懂 | 錯誤處理不良 | 清楚訊息 + 解決建議 |

## 1 個決策點

### 決策情境

**何時需要使用者測試？用什麼方法？**

### 選項對比

| 方法 | 時間 | 成本 | 洞察深度 |
|------|------|------|----------|
| 訪談 | 中 | 中 | 深 |
| 觀察 | 高 | 高 | 很深 |
| 問卷 | 低 | 低 | 淺 |
| 使用性測試 | 中 | 中 | 深 |
| A/B 測試 | 低 | 低 | 中 |

### 決策準則

**探索階段**（不確定問題是什麼）：
- 訪談、觀察

**設計階段**（驗證設計）：
- 使用性測試、原型測試

**優化階段**（改善現有產品）：
- A/B 測試、數據分析

**資源有限時**：
- 5 人使用性測試可發現 85% 問題
- 「游擊式」測試：找同事或朋友

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「我們知道使用者要什麼」症候群**

### 錯誤做法

不做使用者研究，基於假設設計：
- 「我覺得使用者會這樣用」
- 「這個功能很直覺，不需要測試」
- 「我們都是使用者，我們知道」

### 後果

1. **做錯功能**：使用者不需要的功能
2. **做錯介面**：使用者找不到功能
3. **浪費資源**：開發完才發現問題
4. **責任推卸**：「使用者不會用」

### 正確做法

1. **先研究**：訪談、觀察使用者
2. **做原型**：低成本驗證想法
3. **使用性測試**：5 人就夠
4. **持續優化**：數據驅動決策

**經驗法則**：
> 「你不是使用者。」
> 「使用者說的 ≠ 使用者做的。」

## 可落地 Checklist（至少 5 條）

開發功能時用這份清單：

- [x] **狀態回饋**：Loading、成功、錯誤都有回饋嗎？
- [x] **錯誤訊息**：錯誤訊息清楚嗎？有解決建議嗎？
- [x] **鍵盤操作**：可以用鍵盤操作嗎？
- [x] **空狀態**：沒有資料時顯示什麼？
- [x] **一致性**：和其他頁面一致嗎？
- [x] **語意標籤**：用了正確的 HTML 標籤嗎？
- [x] **對比度**：文字對比度夠嗎？

## 章後練習

### 練習：用尼爾森原則評估一個介面

> 目標：評估一個登入頁面

**評估結果**：

| 原則 | 評估 | 改善建議 |
|------|------|----------|
| 系統狀態可見 | ❌ 登入後沒有 Loading | 加 Loading 狀態 |
| 符合真實世界 | ✅ 語言清楚 | - |
| 使用者控制 | ⚠️ 沒有「記住我」選項 | 加選項 |
| 一致性 | ✅ 和其他頁面一致 | - |
| 預防錯誤 | ⚠️ 密碼沒有即時驗證 | 加即時驗證 |
| 辨識優於記憶 | ✅ 有忘記密碼連結 | - |
| 彈性與效率 | ❌ 不能 Enter 提交 | 支援 Enter |
| 簡約設計 | ✅ 只有必要欄位 | - |
| 錯誤識別 | ⚠️ 只說「登入失敗」 | 說明是帳號還是密碼錯 |
| 幫助與文件 | ⚠️ 沒有幫助連結 | 加幫助連結 |

**優先改善**：
1. 加 Loading 狀態（系統狀態可見）
2. 改善錯誤訊息（錯誤識別）
3. 支援 Enter 提交（彈性與效率）

## 延伸思考

### 工程師如何參與 UX？

**我的思考**：

工程師可以做的：

1. **實作時考慮 UX**
   - 每個操作都有回饋
   - 錯誤訊息清楚有用
   - 鍵盤可操作

2. **提出技術限制**
   - 「這個動畫會影響效能」
   - 「這個互動在行動裝置會有問題」

3. **參與使用性測試**
   - 看使用者實際使用
   - 理解使用者痛點

4. **用工具檢查**
   - Lighthouse
   - axe DevTools

### 如何在有限資源下做 UX？

**我的策略**：

1. **最低限度**：
   - 遵循 10 個可用性原則
   - 用設計系統保持一致
   - 做 5 人使用性測試

2. **持續改善**：
   - 收集使用者回饋
   - 看數據找問題
   - 每個 Sprint 改善一點

3. **工程師可以做的**：
   - Code review 時檢查 UX
   - 用 Checklist 自檢
   - 把 UX 問題當 Bug 對待
