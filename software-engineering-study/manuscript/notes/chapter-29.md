## 3 個關鍵概念

### 概念 1：AI 輔助開發

**書中定義**：

使用 AI 工具輔助軟體開發的各個階段。

| 工具 | 功能 |
|------|------|
| GitHub Copilot | 程式碼補全、生成 |
| Claude Code | 編寫、重構、解釋 |
| Cursor | AI-first IDE |

**我的理解**：

AI 是工具，不是替代者。

**AI 改變的**：
- 寫程式碼的方式
- 學習的方式
- 解決問題的效率

**AI 不變的**：
- 需要人審查
- 需要人做決策
- 需要理解系統

**新技能需求**：

| 技能 | 為什麼重要 |
|------|-----------|
| Prompt 工程 | 和 AI 有效溝通 |
| AI 輸出審查 | 識別 AI 錯誤 |
| 系統設計 | AI 不擅長 |
| 問題定義 | 定義問題比解決重要 |

**經驗法則**：
> 「AI 是你的助手，不是你的替代。」

---

### 概念 2：雲原生

**書中定義**：

充分利用雲端優勢的應用設計方式。

| 支柱 | 說明 |
|------|------|
| 容器化 | Docker 打包應用 |
| 微服務 | 拆分為小服務 |
| 動態編排 | Kubernetes 管理 |
| DevOps | 持續交付 |

**我的理解**：

雲原生不只是「放在雲端」，而是「為雲而生」。

| 傳統應用 | 雲原生應用 |
|----------|-----------|
| 單體架構 | 微服務 |
| 手動部署 | 自動化 |
| 垂直擴展 | 水平擴展 |
| 狀態依賴 | 無狀態設計 |

**核心工具**：

| 工具 | 用途 |
|------|------|
| Docker | 容器化 |
| Kubernetes | 編排 |
| Helm | 部署管理 |
| Terraform | 基礎設施即程式碼 |

**經驗法則**：
> 「容器化是基本功，K8s 是必備技能。」

---

### 概念 3：平台工程

**書中定義**：

建立自助式開發者平台，提升開發者體驗。

| 特點 | 說明 |
|------|------|
| 自助服務 | 開發者自己操作 |
| 黃金路徑 | 標準化的最佳實踐 |
| 抽象複雜度 | 隱藏基礎設施細節 |
| 可觀測性 | 內建監控 |

**我的理解**：

平台工程是 DevOps 的進化，重點是開發者體驗。

**DevOps vs. 平台工程**：

| 面向 | DevOps | 平台工程 |
|------|--------|----------|
| 焦點 | 文化 | 產品 |
| 服務 | 協作 | 體驗 |
| 交付 | 實踐 | 平台 |

**經驗法則**：
> 「好的平台讓開發者忘記基礎設施。」

## 1 個決策點

### 決策情境

**該學什麼新技術？**

### 選項對比

| 策略 | 優點 | 缺點 |
|------|------|------|
| 追逐所有新技術 | 不會錯過 | 學不精 |
| 只學成熟技術 | 穩定可靠 | 可能落後 |
| 根據需求選擇 | 務實 | 需要判斷力 |

### 決策準則

| 優先級 | 技術類型 |
|--------|----------|
| 高 | 工作需要的 |
| 高 | 基礎性的（容器、K8s） |
| 中 | 提升效率的（AI 工具） |
| 低 | 炒作中但不成熟的 |

**我的策略**：
1. 先學基礎（容器、K8s）
2. 善用 AI 工具提升效率
3. 關注但不急於學習新炒作
4. 根據工作需求調整

**經驗法則**：
> 「學會使用 AI，而不是和 AI 競爭。」

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「技術追逐」症候群**

### 錯誤做法

不停追逐新技術：
- 「這個新框架要學」
- 「那個語言要學」
- 「這個工具很熱門」
- 結果什麼都學不精

### 後果

1. **疲於奔命**：永遠追不完
2. **淺嚐輒止**：沒有深度
3. **忽略基本功**：原則不扎實
4. **焦慮**：永遠覺得不夠

### 正確做法

**有策略的學習**：

| 做法 | 說明 |
|------|------|
| 基礎優先 | 先學不變的原則 |
| 深度優先 | 一個領域學深 |
| 需求驅動 | 根據工作需求 |
| T 型人才 | 一專多能 |

**選擇技術的問題**：
- 這解決我什麼問題？
- 這是炒作還是真正有價值？
- 我現在需要學嗎？
- 學習成本和回報如何？

**檢驗標準**：
> 「你能用學到的技術解決實際問題嗎？」

## 延伸思考

### AI 會取代軟體工程師嗎？

**我的看法**：

不會取代，但會改變。

**會被影響的**：
- 純粹寫程式碼的工作
- 重複性高的工作
- 簡單的 CRUD 開發

**不會被取代的**：
- 系統設計
- 問題定義
- 業務理解
- 人際溝通
- 判斷和決策

**結論**：
> 「和 AI 協作的工程師會取代不和 AI 協作的工程師。」

### 如何在快速變化中保持競爭力？

**我的策略**：

1. **基礎紮實**
   - 軟體工程原則不變
   - 演算法和資料結構
   - 系統設計

2. **擁抱 AI**
   - 學會使用 AI 工具
   - 和 AI 協作

3. **持續學習**
   - 保持好奇心
   - 有策略地學習

4. **軟技能**
   - 溝通能力
   - 問題解決
   - 領導力
