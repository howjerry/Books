## 章前預測（5 分鐘）

### 這章應該回答什麼問題？

1. 如何根據情境選擇適合的流程？
2. 有沒有一個決策框架可以參考？
3. 混合流程是好還是壞？
4. 流程如何隨團隊演進？

### 我現在的理解是什麼？

閱讀前，我的理解：
- 選流程靠經驗和直覺
- 混合流程 = 妥協 = 不好
- 越多流程 = 越成熟

### 我遇過哪些相關痛點？

1. **選錯流程**：用 Scrum 做合規專案，文件補不完
2. **混合變四不像**：說是敏捷，但有瀑布的簽核，兩邊好處都沒有
3. **流程一成不變**：三年前定的流程，現在還在用，但情境已經變了

---

## 本章一句話

> **選擇流程的關鍵不是「哪個最好」，而是「哪個最適合我的情境」。用六個維度評估，用決策樹選擇，用務實態度混合，用持續改進演化。**

---

## 3 個關鍵概念

### 概念 1：六個選擇維度

**書中定義**：

選擇流程時需要評估六個維度：
1. **需求穩定性**：需求有多穩定？
2. **技術風險**：技術方案有多大不確定性？
3. **回饋可得性**：多容易獲得使用者回饋？
4. **合規要求**：有多嚴格的文件/審計要求？
5. **團隊特性**：規模、分佈、成熟度如何？
6. **專案類型**：全新產品、迭代、整合、維運？

**我的理解**：

這六個維度提供了一個**系統性的評估框架**，而不是靠直覺選流程。

以前我選流程的方式：「大家都用 Scrum，那我們也用吧。」

現在應該這樣想：
1. 我們的需求穩定嗎？→ 不太穩定
2. 技術風險高嗎？→ 中等
3. 能獲得頻繁回饋嗎？→ 可以
4. 有合規要求嗎？→ 有一些
5. 團隊特性？→ 小團隊、共處
6. 專案類型？→ 產品迭代

結論：Scrum 適合，但需要增加必要的文件產出。

---

### 概念 2：混合流程是務實的選擇

**書中定義**：

混合流程不是妥協，而是務實的選擇。常見模式：
- **Water-Scrum-Fall**：需求和部署用瀑布，開發用 Scrum
- **敏捷 + 合規文件**：敏捷開發，但產出必要文件
- **原型 + 增量**：先原型探索，再增量開發
- **Scrumban**：Scrum 節奏 + Kanban 流動

混合的原則：目的清晰、介面明確、一致性、可調整。

**我的理解**：

這解答了我的一個困惑——**混合不是「不夠純粹」，而是「更適應現實」**。

現實中，很少有專案完美符合單一流程：
- 用 Scrum，但客戶要求簽核才能開始
- 用敏捷，但監管要求完整文件
- 開發可以敏捷，但部署有審批流程

**好的混合**：
- 清楚知道為什麼這樣混合
- 每個元素都有明確目的
- 持續評估和調整

**壞的混合**：
- 形式上敏捷，骨子裡瀑布
- 為了「看起來現代」而混合
- 兩邊的壞處都有

---

### 概念 3：最高成熟度是最少但足夠的流程

**書中定義**：

流程成熟度的演進：
1. 初始：混亂、英雄主義
2. 可重複：有流程但依賴個人
3. 已定義：流程標準化
4. 量化管理：用數據管理
5. 持續優化：最少但足夠的流程

最高成熟度不是「最多流程」，而是「最少但足夠」。

**我的理解**：

這打破了一個迷思：**流程多 ≠ 成熟**。

初始階段需要建立流程，是因為團隊還沒有共同的做事方式。

但當團隊足夠成熟時，很多流程變成內化的習慣，不需要明確規定。

**類比**：
- 新手駕駛需要口訣（看鏡子、打方向燈、轉頭確認...）
- 資深駕駛這些都是自動的，不需要口訣

**實際意義**：
- 不要為了「完整」而增加流程
- 定期問「這個流程還需要嗎？」
- 目標是讓做對的事變成自然的事

## 原則、方法、護欄

### 原則（Principle）

| 原則 | 適用情境 | 說明 |
|------|----------|------|
| 情境決定流程 | 選擇流程時 | 不是哪個最好，是哪個最適合 |
| 六維度評估 | 評估專案特性時 | 系統性地分析需求穩定性、風險、回饋等 |
| 混合是務實 | 單一流程不適合時 | 清楚目的、明確介面、保持一致 |
| 漸進演進 | 改進流程時 | 不要一步到位，小步持續改進 |
| 最少但足夠 | 成熟團隊 | 流程多不等於成熟 |

---

### 方法（Practice）

| 方法 | 步驟 | 產物 |
|------|------|------|
| 流程選擇 | 六維度評估 → 決策樹 → 選擇基礎 → 調整 | 適合情境的流程 |
| 混合設計 | 識別需求 → 選擇元素 → 定義介面 → 試運行 | 混合流程定義 |
| 流程演進 | 評估現況 → 識別痛點 → 小步改進 → 驗證效果 | 持續改進的流程 |

---

### 護欄（Guardrail）

| 危險信號 | 可能原因 | 修正方式 |
|----------|----------|----------|
| 所有專案用同一流程 | 沒有考慮情境差異 | 用六維度評估每個專案 |
| 混合變成四不像 | 沒有明確目的 | 回到原則，問「為什麼這樣混合」 |
| 流程越加越多 | 用流程解決人的問題 | 問「這個流程解決什麼問題」 |
| 流程成為負擔 | 過度流程或不適合 | 精簡、調整或更換 |
| 改流程遇到抗拒 | 變化太大、沒有說明原因 | 小步改進、解釋 why |

## 1 個決策點

### 決策情境

**我的專案應該用什麼流程？**

### 快速評估表

| 維度 | 評估問題 | 我的專案 |
|------|----------|----------|
| 需求穩定性 | 需求會變嗎？ | _____ |
| 技術風險 | 技術方案確定嗎？ | _____ |
| 回饋可得性 | 能頻繁獲得回饋嗎？ | _____ |
| 合規要求 | 有嚴格文件要求嗎？ | _____ |
| 團隊特性 | 團隊規模/分佈/成熟度？ | _____ |
| 專案類型 | 新產品/迭代/維運？ | _____ |

### 決策準則

```
需求穩定 + 合規嚴格 → 瀑布/V模型
需求穩定 + 合規寬鬆 → 增量
需求不穩定 + 可回饋 + 低風險 → Scrum/XP
需求不穩定 + 可回饋 + 高風險 → 螺旋 + 敏捷
需求不穩定 + 難回饋 → 增量 + 原型
維運/支援工作 → Kanban
```

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「流程拜物教」症候群**（Process Cargo Cult）

### 錯誤做法

盲目複製其他公司的流程，因為「他們成功了，所以這個流程一定好」：
- Spotify 用 Squad，我們也用 Squad
- Google 用 OKR，我們也用 OKR
- Netflix 沒有流程，我們也不要流程

### 後果

1. **水土不服**：別人的流程不適合我們的情境
2. **形式主義**：有形式沒有精神
3. **責任推卸**：「流程說的」變成擋箭牌
4. **放棄思考**：不問為什麼，只問怎麼做

### 正確做法

1. **理解原理**：這個流程解決什麼問題？為什麼有效？
2. **評估情境**：我們的情境和他們一樣嗎？
3. **適應調整**：根據我們的情境調整
4. **持續驗證**：這個流程對我們有效嗎？

## 可落地 Checklist（至少 5 條）

選擇或評估流程時，用這份清單：

- [x] **評估六維度**：我有系統性地評估專案特性嗎？
- [x] **匹配情境**：選擇的流程適合這個專案的特性嗎？
- [x] **明確目的**：每個流程元素都有明確的目的嗎？
- [x] **混合合理**：如果混合，介面和一致性有考慮嗎？
- [x] **可以調整**：流程可以根據回饋調整嗎？
- [x] **定期評估**：有機制定期評估流程效果嗎？
- [x] **最少足夠**：這是達成目的的最少流程嗎？

### 練習：評估我目前專案的流程適配度

| 維度 | 我的專案特性 | 目前流程 | 匹配度 | 調整建議 |
|------|-------------|---------|--------|----------|
| 需求穩定性 | 中（核心穩定，細節變） | Scrum | 🟢 好 | - |

| 合規要求 | 中（需要審計文件） | 最小文件 | 🔴 差 | 增加必要文件產出 |

| 專案類型 | 產品迭代 | Scrum | 🟢 好 | - |

**總結**：Scrum 基本適合，但需要：
1. 增加真實使用者回饋
2. 在 Sprint 產出中加入審計需要的文件

## 延伸思考

### 如果讓你設計一個「理想的流程」，它會長什麼樣子？

**我的設計**：

```
核心節奏：2 週迭代
├── 迭代開始：規劃 + 風險識別（半天）
├── 每日：15 分鐘同步（站會）
├── 開發中：結對/Mob 編程、TDD、持續整合
├── 迭代中：隨時可 Demo 給 PO
├── 迭代結束：Review + Retro（半天）
└── 每迭代：至少一個可部署的增量

品質內建：
├── 自動化測試（覆蓋率 > 80%）
├── 程式碼審查（PR 必須）
├── 持續整合（每次提交）
└── 持續部署（每天可部署）

文件精簡：
├── ADR（架構決策記錄）
├── 需求追溯（用戶故事 → 程式碼 → 測試）
└── 發布說明

持續改進：
├── 每迭代 Retro 至少一個行動
├── 每季度流程回顧
└── 年度流程健檢
```

### 「最少但足夠的流程」對你的團隊意味著什麼？

**我的思考**：

對我們團隊來說，「最少但足夠」意味著：

**保留**：
- Daily Standup（同步和求助必要）
- Sprint Review（回饋必要）
- Retrospective（改進必要）
- 程式碼審查（品質必要）

**可以簡化**：
- Sprint Planning 可以更短
- 估算可以用 T-shirt size 而非精確點數
- 文件只寫必要的

**可以移除**：
- 過度詳細的 Jira 欄位
- 每週的狀態報告（用看板取代）
- 過時的 Wiki 頁面
