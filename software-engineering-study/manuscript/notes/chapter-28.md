## 3 個關鍵概念

### 概念 1：PDCA 循環

**書中定義**：

Plan → Do → Check → Act 的持續循環。

| 階段 | 活動 |
|------|------|
| Plan | 識別問題、制定計畫 |
| Do | 小規模試行 |
| Check | 評估結果 |
| Act | 標準化或調整 |

**我的理解**：

這是改善的基本框架。

**以前**：出問題 → 臨時修 → 繼續出問題
**現在**：識別 → 試行 → 驗證 → 標準化

**我的實踐**：

改善 PR 審查時間：

| 階段 | 活動 |
|------|------|
| Plan | 問題：審查要 3 天。策略：限制 PR 大小 |
| Do | 試行 2 週 |
| Check | 結果：降到 1.5 天 |
| Act | 寫入規範，進入下一循環 |

**經驗法則**：
> 「改善是循環，不是直線。」

---

### 概念 2：回顧會議（Retrospective）

**書中定義**：

團隊定期反思和改善的會議。

| 階段 | 時間 | 活動 |
|------|------|------|
| 開場 | 5 分鐘 | 設定安全氛圍 |
| 收集 | 15 分鐘 | 蒐集資料 |
| 洞察 | 20 分鐘 | 找出原因 |
| 決定 | 15 分鐘 | 選擇行動 |
| 結束 | 5 分鐘 | 感謝、回饋 |

**我的理解**：

回顧是敏捷中最重要的改善機制。

**回顧的反模式**：

| 反模式 | 解法 |
|--------|------|
| 沒有行動 | 明確行動項目和負責人 |
| 同樣問題 | 追蹤改善成效 |
| 只有批評 | 平衡正面和負面 |
| 責備 | 專注系統改善 |

**我的實踐**：

使用 Start-Stop-Continue：
- Start：開始做什麼？
- Stop：停止做什麼？
- Continue：繼續做什麼？

**經驗法則**：
> 「沒有行動的回顧只是抱怨會。」

---

### 概念 3：Kaizen 持續改善

**書中定義**：

每天進行小改善的文化。

| 原則 | 說明 |
|------|------|
| 持續 | 每天都在改善 |
| 漸進 | 小步快跑 |
| 全員 | 每個人都參與 |
| 現場 | 在工作現場改善 |

**我的理解**：

改善不是大事件，而是日常習慣。

**大改革 vs. Kaizen**：

| 面向 | 大改革 | Kaizen |
|------|--------|--------|
| 規模 | 大 | 小 |
| 頻率 | 偶爾 | 每天 |
| 風險 | 高 | 低 |
| 參與 | 少數人 | 全員 |
| 抗拒 | 大 | 小 |

**經驗法則**：
> 「每天進步 1%，一年後是 37 倍。」

## 原則、方法、護欄

### 流程改善工具箱

| 工具 | 用途 | 適用 |
|------|------|------|
| PDCA | 任何改善 | 通用 |
| 5 Whys | 根因分析 | 問題分析 |
| 魚骨圖 | 原因分析 | 複雜問題 |

| 價值流圖 | 找浪費 | 流程優化 |

---

### 護欄（Guardrail）

| 危險信號 | 可能的問題 | 修正方式 |
|----------|-----------|----------|
| 沒有回顧 | 錯失改善機會 | 定期回顧 |
| 回顧沒行動 | 浪費時間 | 追蹤行動 |
| 同樣問題 | 沒有根本解決 | 根因分析 |
| 只有管理層改善 | 脫離現場 | 全員參與 |
| 改善沒數據 | 不知道有沒有效 | 度量驅動 |

## 1 個決策點

### 決策情境

**大改革 vs. 漸進改善？**

### 選項對比

| 選項 | 優點 | 缺點 |
|------|------|------|
| 大改革 | 快速改變 | 風險高、抗拒大 |
| 漸進改善 | 風險低、抗拒小 | 速度慢 |
| 混合 | 平衡 | 需要判斷 |

### 決策準則

| 情境 | 建議 |
|------|------|
| 危機、必須快速改變 | 大改革 |
| 日常改善 | 漸進 |
| 有時間、有空間 | 漸進 |
| 文化變革 | 漸進 + 耐心 |

**我的策略**：
- 預設選擇漸進改善
- 除非有緊迫理由，否則不做大改革
- 小步快跑，快速迭代

**經驗法則**：
> 「漸進改善是預設，大改革是例外。」

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「改革疲勞」症候群**

### 錯誤做法

持續推動改革，但沒有成效：
- 「我們又要改了」
- 「這次一定會成功」
- 「為什麼大家不配合」

### 後果

1. **信任流失**：團隊不再相信改革
2. **士氣低落**：又來了的心態
3. **抗拒增加**：越改越抗拒
4. **效果遞減**：改革越來越難

### 正確做法

**確保改善有成效**：

| 做法 | 說明 |
|------|------|
| 小規模試行 | 先在小範圍試 |
| 快速見效 | 選擇容易成功的開始 |
| 慶祝勝利 | 讓大家看到成果 |
| 追蹤成效 | 用數據證明 |

**改善前問自己**：
- 這個改善解決什麼問題？
- 我們怎麼知道成功了？
- 誰會抗拒？為什麼？
- 我們有資源完成嗎？

**檢驗標準**：
> 「團隊期待下一次改善嗎？」

## 章後練習

### 練習：規劃一個改善循環

> 目標：用 PDCA 規劃一個實際的改善

**我選擇的改善主題**：減少部署失敗率

**PDCA 計畫**：

| 階段 | 活動 | 產出 |
|------|------|------|
| Plan | 分析最近 10 次部署失敗原因 | 原因分類 |
| Do | 針對最常見原因，加入自動化檢查 | 檢查腳本 |
| Check | 觀察下一個月的失敗率 | 趨勢圖 |
| Act | 如果有效，標準化；否則調整策略 | 流程更新 |

**成功指標**：
- 部署失敗率從 15% 降到 5%

**時間規劃**：
- Plan：1 週
- Do：2 週
- Check：4 週
- Act：1 週

## 延伸思考

### 如何讓團隊願意持續改善？

**我的策略**：

1. **心理安全**
   - 不懲罰報告問題的人
   - 專注系統，不責備個人

2. **快速見效**
   - 選擇容易成功的開始
   - 慶祝小勝利

3. **透明追蹤**
   - 讓改善進度可見
   - 分享成功案例

4. **賦權團隊**
   - 讓團隊自己決定改善什麼
   - 給予資源和時間

### CMMI 在敏捷時代還有價值嗎？

**我的看法**：

有，但要正確使用。

**CMMI 的價值**：
- 提供成熟度評估框架
- 識別改善機會
- 提供通用語言

**敏捷時代的 CMMI**：
- 輕量化採用
- 重視實質而非形式
- 和敏捷實踐結合

**結論**：
CMMI 是工具，不是目的。用它來識別改善機會，而不是為了證書。
