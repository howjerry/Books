## Part 1 核心收穫

經過這一部分的學習，我得到以下核心認知：

### 1. 流程是工具，不是宗教

流程的目的是降低風險、提高效率、確保品質——而非「遵守流程」本身。

### 2. 選擇流程要看情境

沒有「最佳」流程，只有「最適合情境」的流程。用六個維度評估，而非跟風。

### 3. 敏捷的本質是回饋迴路

敏捷的形式（Sprint、Standup）不是目的，縮短回饋迴路才是目的。

### 4. 人的問題比技術問題更致命

專案失敗的主因是溝通，不是技術。心理安全是高效能團隊的核心。

## 必做輸出一：流程模型選擇矩陣

### 六維度評估表

在選擇流程前，先評估專案的六個維度：

| 維度 | 評估問題 | 低 | 中 | 高 |
|------|----------|----|----|----|
| **需求穩定性** | 需求有多穩定？ | 會持續變化 | 核心穩定，細節變 | 可完全確定 |
| **技術風險** | 技術方案確定嗎？ | 成熟技術 | 部分新技術 | 大量不確定 |
| **回饋可得性** | 能獲得使用者回饋嗎？ | 困難 | 定期可得 | 隨時可得 |
| **合規要求** | 文件/審計要求多嚴格？ | 最小文件 | 需要關鍵文件 | 完整文件 |
| **團隊規模** | 團隊多大？ | < 7 人 | 7-15 人 | > 15 人 |
| **團隊分佈** | 團隊共處還是分散？ | 共處 | 部分分散 | 高度分散 |

### 決策矩陣

根據評估結果，選擇適合的流程：

| 情境組合 | 推薦流程 | 原因 |
|----------|----------|------|
| 需求高穩定 + 高合規 | **瀑布 / V 模型** | 需要完整文件、審計軌跡 |
| 需求高穩定 + 低合規 | **增量模型** | 分批交付，但不需太多文件 |
| 需求低穩定 + 高回饋 + 小團隊 | **Scrum / XP** | 短迭代、快速回饋 |
| 需求低穩定 + 高回饋 + 高風險 | **螺旋 + 敏捷** | 風險驅動 + 迭代 |
| 需求低穩定 + 低回饋 | **增量 + 原型** | 用原型驗證，增量交付 |
| 維運/支援工作 | **Kanban** | 流動效率、處理插件 |
| 大規模團隊 | **SAFe / LeSS** | 跨團隊協調 |

### 決策流程圖

```
開始
  │
  ▼
需求穩定嗎？
  │
  ├─ 是 ──► 有嚴格合規要求嗎？
  │            │
  │            ├─ 是 ──► 【瀑布 / V 模型】
  │            │
  │            └─ 否 ──► 【增量模型】
  │
  └─ 否 ──► 能獲得頻繁回饋嗎？
               │

               │            │

               │            │
               │            └─ 否 ──► 團隊大小？
               │                        │
               │                        ├─ 小 ──► 【Scrum / XP】
               │                        │
               │                        └─ 大 ──► 【SAFe / LeSS】
```

## 必做輸出二：團隊行為護欄

### 健康團隊的可觀察行為

| 維度 | 健康行為 | 觀察指標 |
|------|----------|----------|
| **心理安全** | 敢說真話、承認錯誤 | 會議上有人提出疑問、Retro 有真實問題 |
| **溝通** | 確認理解、主動同步 | 少有「我以為你知道」的誤解 |
| **衝突** | 對事不對人、建設性爭論 | 技術爭論有結論、不影響關係 |
| **承諾** | 信守承諾、主動回報 | 說到做到、困難提早說 |
| **協作** | 互相幫助、知識共享 | 不是「這不是我的事」 |
| **改進** | 定期反思、落實行動 | Retro 有行動項且有 follow-up |

### 危險信號與應對

| 危險信號 🚩 | 可能原因 | 應對方式 |
|------------|----------|----------|
| 沒人敢提問題 | 怪罪文化、不安全 | 領導者示範、感謝提問 |
| 會議一片和諧，會後抱怨 | 假和諧、不敢表達 | 創造匿名發言機制 |
| 技術爭論變人身攻擊 | 缺乏衝突處理技巧 | 建立爭論規則、調解者 |
| 總是同一個人救火 | 英雄文化、知識集中 | 知識共享、結對/Mob |
| 壞消息總是太晚 | 怪罪文化、報喜不報憂 | Blameless 文化 |
| 承諾經常跳票 | 過度承諾、估算問題 | 學習說「不」、改進估算 |
| Retro 沒有行動 | 形式主義、無力改變 | 每次至少一個改變 |
| 團隊停滯不前 | 缺乏挑戰、沒有成長 | 引入新實踐、輪調 |

### 建立健康文化的實踐

**日常實踐**：
- [ ] Daily Standup：同步、求助、不報告
- [ ] 程式碼審查：學習、分享、不批判
- [ ] 結對/Mob 編程：知識共享、即時回饋

**定期實踐**：
- [ ] Sprint Retrospective：每兩週反思、至少一個改變
- [ ] Blameless Postmortem：出問題時聚焦學習
- [ ] 團隊健康檢查：每季度評估團隊狀態

**領導者行為**：
- [ ] 承認自己的錯誤
- [ ] 感謝（不責備）帶來壞消息的人
- [ ] 失敗時問「學到什麼」，不問「誰的錯」
- [ ] 鼓勵實驗，允許失敗

## 關鍵問題清單

在專案開始或評估流程時，問自己：

### 關於流程

1. 我的需求穩定嗎？變化頻率如何？
2. 技術風險高嗎？有多少不確定性？
3. 能多快獲得真正的使用者回饋？
4. 有什麼合規/文件要求必須滿足？
5. 團隊規模和分佈如何？
6. 這是什麼類型的專案（新產品/迭代/維運）？

### 關於團隊

7. 團隊有心理安全嗎？敢說真話嗎？
8. 溝通有效嗎？常有誤解嗎？
9. 衝突如何處理？對事還是對人？
10. 有持續改進的機制嗎？
11. 領導風格適合團隊成熟度嗎？
