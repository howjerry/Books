## 章前預測（5 分鐘）

### 這章應該回答什麼問題？

1. 專案管理是什麼？
2. 專案經理的角色是什麼？
3. 如何管理利害關係人？
4. 如何帶領團隊？

### 我現在的理解是什麼？

閱讀前，我的理解：
- PM = 排程、開會、追進度
- PM 要讓所有人開心
- PM 要精通技術

### 我遇過哪些相關痛點？

1. **範圍蔓延**：功能越做越多，時間不變
2. **溝通不良**：不同人期待不同
3. **團隊衝突**：成員之間有摩擦

---

## 本章一句話

> **專案管理是在範圍、時間、成本的限制下，帶領團隊完成目標。核心職責是規劃、組織、領導、控制和溝通。管理利害關係人期待，帶領團隊經歷 Forming → Storming → Norming → Performing。**

---

## 3 個關鍵概念

### 概念 1：鐵三角

**書中定義**：

```
範圍 ←→ 時間 ←→ 成本
```

改變一個，必須調整其他。

**我的理解**：

這是專案管理的基本限制。

**常見情況**：

| 情況 | 結果 |
|------|------|
| 固定範圍 + 固定時間 | 品質下降或成本增加 |
| 固定範圍 + 固定成本 | 時間延長 |
| 固定時間 + 固定成本 | 範圍縮減 |

**我的經驗**：

老闆說：「這個功能一定要做，時間不能變，人也不能加。」
結果：品質下降，技術債累積。

**正確做法**：

明確哪個是可以調整的：
- 「如果要在時間內完成，範圍需要調整」
- 「如果範圍不變，需要更多資源或時間」

**經驗法則**：
> 「沒有三個都固定的專案，只有沒說清楚的專案。」

---

### 概念 2：利害關係人矩陣

**書中定義**：

根據「利益」和「影響力」分類利害關係人。

| 高影響 + 高利益 | 密切管理 |
| 高影響 + 低利益 | 保持知情 |
| 低影響 + 高利益 | 保持滿意 |
| 低影響 + 低利益 | 監控 |

**我的理解**：

不是所有人都一樣重要，要策略性溝通。

**我的利害關係人分析**：

| 角色 | 利益 | 影響 | 策略 |
|------|------|------|------|
| 老闆 | 高 | 高 | 每週報告 |
| 用戶 | 高 | 中 | 每週 Demo |
| 團隊 | 高 | 中 | 每日站會 |
| 運維 | 中 | 中 | 里程碑同步 |
| 法務 | 低 | 高 | 關鍵時刻諮詢 |

**經驗法則**：
> 「找出真正能說 Yes/No 的人，確保他們知情。」

---

### 概念 3：團隊發展階段

**書中定義**：

Tuckman 模型：

1. **Forming**（形成）：禮貌、觀望
2. **Storming**（風暴）：衝突、競爭
3. **Norming**（規範）：建立規範、合作
4. **Performing**（高績效）：高效、自主

**我的理解**：

團隊衝突是正常的，是成長的一部分。

**我的經驗**：

新團隊前幾週很和諧，然後開始有摩擦，我以為是問題。
現在理解這是 Storming 階段，需要處理而不是迴避。

**PM 在各階段的角色**：

| 階段 | PM 做什麼 |
|------|-----------|
| Forming | 指導、建立規則、說清楚目標 |
| Storming | 調解衝突、促進討論、不要迴避 |
| Norming | 支持、授權、讓團隊建立自己的方式 |
| Performing | 移除障礙、信任團隊 |

**經驗法則**：
> 「團隊衝突不是問題，迴避衝突才是。」

## 原則、方法、護欄

### PM 職責速查表

| 職責 | 活動 | 產出 |
|------|------|------|
| 規劃 | 定義範圍、估算、排程 | 專案計畫 |
| 組織 | 建立團隊、分配角色 | 組織圖、RACI |
| 領導 | 激勵、決策、解決衝突 | 團隊共識 |
| 控制 | 追蹤進度、管理變更 | 狀態報告 |
| 溝通 | 報告、會議、談判 | 會議記錄 |

---

### 護欄（Guardrail）

| 危險信號 | 可能的問題 | 修正方式 |
|----------|-----------|----------|
| 範圍一直加 | 範圍蔓延 | 變更控制流程 |
| 利害關係人不滿 | 溝通不足 | 更頻繁溝通 |
| 團隊持續衝突 | 卡在 Storming | 主動處理衝突 |
| 進度看不到 | 缺少可見性 | 定期報告、Demo |
| 問題都來找 PM | PM 成為瓶頸 | 授權、培養團隊 |

## 1 個決策點

### 決策情境

**專案超支/超時，怎麼辦？**

### 選項對比

| 選項 | 優點 | 缺點 |
|------|------|------|
| 縮減範圍 | 維持時間和成本 | 功能減少 |
| 延長時間 | 維持範圍和品質 | 延遲交付 |
| 增加資源 | 可能加速 | 成本增加、新人學習曲線 |
| 降低品質 | 維持時間 | 技術債、風險 |

### 決策準則

| 情境 | 建議 |
|------|------|
| 有硬性截止日 | 縮減範圍 |
| 範圍不可變 | 延長時間或增加資源 |
| 預算有限 | 縮減範圍或延長時間 |
| 品質關鍵 | 不要降低品質 |

**經驗法則**：
> 「寧可交付少一點但完整的功能，也不要交付多但半成品。」

## 1 個反模式（Anti-pattern）

### 反模式名稱

**「Yes Man」PM**

### 錯誤做法

對所有需求都說 Yes：
- 「好，我們可以加這個功能」
- 「好，時間不變沒問題」
- 「好，我們會想辦法」

### 後果

1. **範圍蔓延**：功能越來越多
2. **團隊過載**：做不完
3. **品質下降**：趕工
4. **失去信任**：無法交付承諾

### 正確做法

**說 Yes 之前，說清楚影響**：

| 錯誤 | 正確 |
|------|------|
| 「好，我們加這個」 | 「可以，但需要延後 A 功能」 |
| 「沒問題」 | 「可以，但需要增加兩週」 |
| 「我們會想辦法」 | 「讓我評估後回覆」 |

**檢驗標準**：
> 「我說 Yes 的時候，有說清楚代價嗎？」

## 章後練習

### 練習：分析你的專案

> 目標：用本章框架分析目前的專案

**鐵三角分析**：

| 限制 | 狀態 | 可調整？ |
|------|------|----------|
| 範圍 | 已定義，但常有新需求 | ⚠️ 需要控制 |
| 時間 | 固定截止日 | ❌ 不可調 |
| 成本 | 固定團隊 | ❌ 不可調 |

**結論**：只有範圍可調，需要嚴格的變更控制。

**利害關係人分析**：

| 角色 | 影響 | 利益 | 目前溝通 | 改善 |
|------|------|------|----------|------|
| 老闆 | 高 | 高 | 每月 | 每兩週 |
| 客戶 | 高 | 高 | 有問題才找 | 每週 Demo |
| 團隊 | 中 | 高 | 每日 | 維持 |

**團隊狀態**：
- 目前階段：Norming
- 已建立工作規範
- 偶有小摩擦但可處理

## 延伸思考

### 技術人員轉 PM 的挑戰是什麼？

**我的觀察**：

| 工程師思維 | PM 思維 |
|------------|---------|
| 解決技術問題 | 解決人的問題 |
| 追求完美 | 追求夠好 |
| 專注深度 | 關注廣度 |
| 個人貢獻 | 團隊貢獻 |
| 確定性 | 管理不確定性 |

**最大挑戰**：
- 放手讓別人做技術決定
- 接受「夠好」而非「完美」
- 花時間溝通而非寫程式

### 敏捷團隊還需要 PM 嗎？

**我的看法**：

角色可能不叫 PM，但職責還是需要：
- Scrum Master：移除障礙、促進流程
- Product Owner：管理範圍、和利害關係人溝通
- Tech Lead：技術決策

**差異**：
- 傳統 PM：控制導向
- 敏捷 PM：服務導向
