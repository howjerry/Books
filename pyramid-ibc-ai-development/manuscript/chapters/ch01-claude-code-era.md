## 1.1 2026 年的開發現實

讓我們先做一個小測驗。

### 🧪 快速自我檢測

請誠實回答以下問題：

1. 你每天使用 AI 輔助開發工具的時間大約是？
   - [ ] A. 還沒開始用
   - [ ] B. 偶爾用一下（< 1 小時）
   - [ ] C. 經常使用（1-4 小時）
   - [ ] D. 幾乎整天都在用

2. 使用 AI 工具時，你最常遇到的問題是？
   - [ ] A. 不知道怎麼開始
   - [ ] B. AI 產出的程式碼需要大幅修改
   - [ ] C. AI 似乎不理解我的專案架構
   - [ ] D. 其實用得還不錯

3. 你對「指揮 AI」的信心程度是？
   - [ ] A. 完全沒信心
   - [ ] B. 有點信心，但常常失敗
   - [ ] C. 大部分時候還行
   - [ ] D. 我已經是 AI 指揮大師了

如果你的答案大多是 B 或 C，恭喜你——你正處於**「AI 使用者」的典型困境**。

根據 Stack Overflow 2025 年度調查，超過 **68%** 的開發者表示「不知道如何有效指揮 AI」。這不是你的問題，這是整個產業的問題。

> 📚 **深度閱讀**
>
> Stack Overflow 2025 開發者調查報告顯示：
> - 78% 的開發者使用某種形式的 AI 輔助開發工具
> - 但只有 23% 認為自己「充分發揮」了這些工具的潛力
> - 最大的障礙？**「不知道怎麼描述我想要什麼」**

### 問題的根源

讓我們用一個比喻來說明這個情況。

**想像你要指揮一個超級聰明的實習生。**

這個實習生（就叫他小克吧）有以下特點：
- 他讀過所有的程式設計書籍
- 他見過數十億行程式碼
- 他可以在幾秒內寫出上百行程式碼
- 他非常聽話，你說什麼他就做什麼

聽起來很棒對吧？但問題是：

- 他不知道你的專案長什麼樣子（除非你告訴他）
- 他不知道你們公司的 coding style（除非你告訴他）
- 他不知道你想達成什麼目標（除非你告訴他）
- 他會用他認為「最好」的方式來做事（這不一定是你需要的方式）

**小克的能力不是問題，你跟小克溝通的方式才是問題。**

## 1.2 從「程式員」到「AI 工作流程架構師」

這裡有一個關鍵的思維轉變。

### 舊思維 vs 新思維

```
【舊思維：程式員】
你 → 寫程式碼 → 產品

【新思維：AI 工作流程架構師】
你 → 設計意圖和約束 → AI → 寫程式碼 → 你 Review → 產品
```

> 🎯 **實戰技巧**
>
> 「AI 工作流程架構師」的核心工作不是寫程式碼，而是：
> 1. **設計意圖**：清楚表達你想要什麼
> 2. **定義約束**：說明什麼可以做、什麼不可以做
> 3. **提供上下文**：給 AI 足夠的資訊來做決策
> 4. **品質把關**：Review AI 的產出並給予回饋

讓我們看一個具體的例子。

### 😰 糟糕的指令

```
幫我寫一個用戶登入功能
```

這個指令有什麼問題？幾乎什麼都沒說清楚：
- 用什麼語言？什麼框架？
- 要支援哪些登入方式？帳號密碼？OAuth？
- 資料庫結構是什麼？
- 有沒有現有的認證系統？
- 前端還是後端？還是全端？

### 😊 好的指令

```
我需要在現有的 FastAPI 專案中新增用戶登入 API。

【意圖】
作為一個用戶，我希望能夠用 email 和密碼登入系統，
這樣我就可以存取受保護的資源。

【行為規格】
- POST /api/v1/auth/login
- 輸入：{ "email": string, "password": string }
- 成功輸出：{ "access_token": string, "token_type": "bearer" }
- 錯誤情況：
  - 401: email 不存在或密碼錯誤
  - 422: 輸入格式不正確

【上下文】
- 使用現有的 User 模型（models/user.py）
- 密碼已用 bcrypt 加密儲存
- JWT token 使用 python-jose，secret 在環境變數 JWT_SECRET
- 現有的路由結構在 routers/ 目錄下
```

看出差別了嗎？

第二個指令像是一份**清楚的工作說明書**。它告訴 AI：
1. **意圖**：為什麼要做這件事（用戶要能登入）
2. **行為**：怎樣算「做完了」（API 規格）
3. **上下文**：在什麼環境下做（專案結構、現有程式碼）

> ⚠️ **注意**
>
> 這不是要你每次都寫這麼長的指令！這是要幫助你理解**好指令的結構**。當你熟練之後，你會知道什麼情況需要說明什麼。就像學開車一樣，一開始你需要想每一個動作，但熟練後就會變成直覺。

## 1.3 Claude Code 的三大核心能力

好，現在你理解了為什麼需要改變思維。讓我們來認識一下你的新夥伴——Claude Code。

Claude Code 不只是一個「會寫程式的聊天機器人」。它是一個**完整的開發環境助手**，具備三大核心能力：

### 能力一：檔案系統整合

```
┌─────────────────────────────────────────┐
│  你的專案目錄                            │
│  ├── src/                               │
│  │   ├── models/        ← Claude 可以讀 │
│  │   ├── routers/       ← Claude 可以讀 │
│  │   └── services/      ← Claude 可以讀 │
│  ├── tests/             ← Claude 可以讀 │
│  └── pyproject.toml     ← Claude 可以讀 │
└─────────────────────────────────────────┘
```

**簡單來說...**

Claude Code 可以直接「看到」你的專案結構。它不只是在一個對話框裡猜測你的程式碼長什麼樣子——它可以實際讀取、理解、甚至修改你的檔案。

**深入一點...**

這意味著你不需要手動複製貼上程式碼。你可以直接說：

```
請看一下 src/models/user.py 的結構，
然後在 src/routers/ 新增一個 auth.py 來處理認證相關的路由
```

Claude Code 會：
1. 讀取 `user.py` 理解你的資料模型
2. 觀察 `routers/` 目錄下其他檔案的風格
3. 建立符合你專案風格的 `auth.py`

> 💡 **小提示**
>
> 第一次使用 Claude Code 時，它會請求存取你的專案目錄的權限。請確保你信任這個操作——Claude Code 確實會讀取和修改你的檔案。

### 能力二：終端機整合

Claude Code 可以執行終端機指令。這不是說它會「建議」你執行什麼指令——它可以**直接執行**。

```
你：請幫我執行測試，看看目前的測試覆蓋率

Claude Code：好的，讓我執行測試...

$ pytest --cov=src tests/
============================= test session starts ==============================
...
---------- coverage: 3.0.0 -----------
Name                    Stmts   Miss  Cover
-------------------------------------------
src/models/user.py         45      3    93%
src/routers/users.py       62     12    81%
src/services/auth.py       38      8    79%
-------------------------------------------
TOTAL                     145     23    84%

你的整體測試覆蓋率是 84%。我注意到 auth.py 的覆蓋率較低，
主要是因為錯誤處理的分支沒有被測試到。要我幫你補充這些測試案例嗎？
```

**這代表什麼？**

你可以讓 Claude Code 幫你：
- 執行測試並分析結果
- 安裝套件
- 執行 linter 和 formatter
- 啟動開發伺服器
- 執行 git 指令
- 任何你平常在終端機做的事

> ⚠️ **注意**
>
> 權力越大，責任越大。Claude Code 可以執行終端機指令，這也意味著它可能執行有風險的操作。好消息是，預設情況下它會在執行敏感操作前詢問你的確認。

### 能力三：MCP 整合

MCP（Model Context Protocol）是 Claude Code 的超能力擴充系統。

**想像一下...**

你正在修一個 bug。這個 bug 是用戶在 Linear 上回報的。修完 bug 後，你需要更新 Linear 的 ticket 狀態，然後在 Slack 通知 QA 團隊可以測試了。

**傳統流程：**
1. 打開 Linear，找到 ticket，複製 bug 描述
2. 回到 IDE，開始修 bug
3. 修完後，切回 Linear，更新狀態
4. 打開 Slack，找到 QA 頻道，發送訊息

**使用 Claude Code + MCP：**
```
你：幫我看一下 Linear ticket LIN-1234 的內容，
    然後修復這個 bug，
    修完後把 ticket 狀態改成 "Ready for QA"，
    並在 #qa-team Slack 頻道通知他們

Claude Code：好的，讓我處理這些事情...

[讀取 Linear ticket]
這是一個用戶回報的登入問題，當 email 包含 "+" 符號時會登入失敗...

[分析並修復 bug]
我找到問題了，在 src/services/auth.py 的 email 驗證邏輯有誤...
已修復並新增對應的測試案例。

[更新 Linear]
已將 LIN-1234 狀態更新為 "Ready for QA"

[通知 Slack]
已在 #qa-team 發送訊息：
"LIN-1234 (Email 含 + 符號無法登入) 已修復，請協助測試 🙏"
```

**這就是 MCP 的威力**——它讓 Claude Code 可以連接到你工作中使用的各種工具。

## 1.4 你的第一個 Claude Code 任務

好了，理論說得夠多了。**讓我們動手做點什麼！**

### 🚀 試試看

如果你還沒有安裝 Claude Code，請先參考附錄 A 完成安裝。

安裝好之後，讓我們來做一個簡單的練習。

### 任務：建立一個 Hello World API

**步驟 1：開啟終端機，進入一個空的資料夾**

```bash
mkdir my-first-claude-project
cd my-first-claude-project
```

**步驟 2：啟動 Claude Code**

```bash
claude
```

**步驟 3：給 Claude Code 你的第一個任務**

複製並貼上以下指令：

```
請幫我建立一個最簡單的 FastAPI 專案，包含：

【意圖】
我想要有一個可以運行的 API 伺服器，
作為之後學習 AI 輔助開發的起點。

【行為規格】
- GET / 回傳 { "message": "Hello, World!" }
- GET /health 回傳 { "status": "healthy" }

【技術要求】
- 使用 Python 3.11+
- 使用 FastAPI
- 使用 uv 作為套件管理工具
- 建立 pyproject.toml
- 包含一個基本的 README.md

請一步一步執行，每一步都告訴我你在做什麼。
```

**步驟 4：觀察 Claude Code 的回應**

Claude Code 會開始：
1. 建立專案結構
2. 建立 `pyproject.toml`
3. 建立 `main.py`
4. 建立 `README.md`
5. 安裝相依套件
6. （可能）啟動伺服器讓你測試

> 💡 **小提示**
>
> 當 Claude Code 要執行終端機指令或修改檔案時，它會先問你是否同意。一開始建議你都選擇「是」來觀察它的行為。等你熟悉之後，可以調整權限設定讓流程更順暢。

### 任務完成檢查

完成後，你應該會看到類似這樣的專案結構：

```
my-first-claude-project/
├── pyproject.toml
├── README.md
└── main.py
```

試著在終端機執行：

```bash
uv run uvicorn main:app --reload
```

然後打開瀏覽器訪問 `http://localhost:8000`，你應該會看到：

```json
{"message": "Hello, World!"}
```

**恭喜！你剛剛完成了第一個由 AI 協作完成的專案！**

- **要點 1**：2026 年，超過 68% 的開發者不知道如何有效指揮 AI。問題不在工具，在於溝通方式。

- **要點 2**：你需要從「程式員」轉變為「AI 工作流程架構師」。你的工作是設計意圖、定義約束、提供上下文、品質把關。
